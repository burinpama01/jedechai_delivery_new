Codebase Issue Audit

รายงานการรีวิวข้อบกพร่องและประเด็นปัญหาในระบบ

วิเคราะห์โครงสร้างโค้ด สถาปัตยกรรม และช่องโหว่ความปลอดภัยของระบบ Jedechai Delivery

เมนูนำทาง

1. สรุปผลการตรวจสอบ (Audit Executive Summary)

จากการรีวิวและสแกน Source Code ของระบบ Jedechai Delivery (Flutter App, Admin Web, และ Supabase Edge Functions/Database) เพื่อตรวจสอบข้อบกพร่อง ทางทีมพัฒนาได้ระบุประเด็นปัญหาหลักที่ต้องดำเนินการแก้ไขจำนวนทั้งสิ้น 11 ประเด็น โดยมีรายละเอียดภาพรวมดังนี้:

Critical Issues
3 จุด
ความเสี่ยงระดับวิกฤต (ต้องแก้ทันที)
High Issues
2 จุด
ส่งผลต่อฟังก์ชันหลักและข้อมูล
Medium Issues
4 จุด
ข้อบกพร่องการแสดงผล / UX
Low Issues
2 จุด
โค้ดที่ไม่ได้ใช้ / คลีนนิ่งโค้ด

สรุปสถาปัตยกรรมปัญหาแยกตามโมดูล:

  • ฟีเจอร์การส่งพัสดุ (Parcel Delivery): พบปัญหามากที่สุด เนื่องจากกระบวนการตั้งแต่การจองฝั่งลูกค้า (Customer UI), การคำนวณราคา, ระบบติดตาม (Tracking UI) ไปจนถึงการรับงานของคนขับ (Driver Navigation) ขาดความต่อเนื่องของสถานะและมี Dead Code ที่สำคัญ
  • แผงควบคุมแอดมิน (Admin Web): มีช่องโหว่การข้ามขั้นตอนการอนุมัติ (Optimistic Lock bypass) เนื่องจากความแตกต่างของวิธีการสโคปตัวแปรของ JavaScript แบบ Classic Script บนเว็บเบราว์เซอร์
  • การแจ้งเตือนและการจัดการทรัพยากร: มีการเรียกใช้งานฟังก์ชันส่งอีเมลโดยไม่มีระบบควบคุมปริมาณ (Rate Limiting) และขาดการกำจัดทรัพยากรขยะใน Storage

2. ประเด็นปัญหาระดับวิกฤต (Critical Issues)

ประเด็นปัญหาที่ส่งผลให้ระบบงานหลักไม่ทำงานตามข้อกำหนด หรือมีช่องโหว่ความปลอดภัยระดับร้ายแรง:

ISSUE-1: โค้ดยืนยันการส่งพัสดุไม่ทำงาน (Driver Confirmation Dead Code) Critical
ไฟล์ที่เกี่ยวข้อง: driver_parcel_confirmation_screen.dart | driver_navigation_screen.dart

อาการ: ลูกค้าจะไม่สามารถรับความคืบหน้าของพัสดุได้จริง เนื่องจากหน้าจอยืนยัน DriverParcelConfirmationScreen ถูกเขียนไว้แต่ไม่มีการนำทาง (Navigate) จากหน้าจอนำทางของคนขับเลย และฟังก์ชันอัปเดตสถานะขนส่ง updateInTransit() ก็ไม่ถูกเรียกจากจุดใดในระบบ ส่งผลให้สถานะพัสดุค้างอยู่ที่ created เสมอ ไม่มีการบันทึกภาพถ่ายการรับของ ส่งของ หรือลายเซ็นผู้รับ

สาเหตุ: ในหน้าจอ driver_navigation_screen.dart มีการข้ามขั้นตอนและใช้ BookingStatusPolicy.driverNextStatus() อัปเดตสถานะตรงๆ โดยไม่มีการดักกรอง (Intercept) เพื่อเปิดหน้าจอยืนยันพัสดุเสียก่อน

แนวทางแก้ไข: เพิ่มเงื่อนไข Interceptor ใน _handleActionPress() สำหรับออเดอร์ประเภทพัสดุ เมื่อสถานะเป็น arrived หรือ in_transit ให้บังคับเปิดหน้าจอยืนยันและอัปโหลดรูปภาพก่อนอัปเดตสถานะ

ISSUE-2: ระบบล็อคการรับออเดอร์แทนร้านค้าถูกข้าม (Optimistic Lock Bypass in Admin Web) Critical
ไฟล์ที่เกี่ยวข้อง: ordersActions.js | app.legacy.js

อาการ: แอดมินสามารถกดยืนยันหรือแทรกแซงออเดอร์ของร้านค้าที่สถานะใดก็ได้โดยไม่มีการป้องกันการกดซ้ำซ้อน ส่งผลให้เกิดปัญหาข้อมูลชนกัน (Race Condition)

สาเหตุ: โค้ดใน ordersActions.js ดึงค่า globalThis.ADMIN_MERCHANT_ACCEPT_STATUSES ไปใช้งาน แต่เนื่องจากตัวแปรนี้ถูกกำหนดด้วยคำสั่ง const ในสคริปต์แบบ Classic (Non-module) ในไฟล์ app.legacy.js ทำให้มันมีขอบเขตภายใน (Lexical Scope) และไม่ได้ถูก Expose ไปยัง globalThis/window ผลลัพธ์คือตัวแปรมีค่าเป็น undefined และระบบยกเลิกการใช้ฟิลเตอร์ป้องกันการอัปเดตทับซ้อนไปโดยปริยาย

แนวทางแก้ไข: ประกาศผูกตัวแปรเข้ากับ globalThis หรือ window ตรงๆ ในจุดเริ่มต้นของระบบ เช่น globalThis.ADMIN_MERCHANT_ACCEPT_STATUSES = [...]

ISSUE-3: ความเสี่ยงการถูกยิงระเบิดอีเมล (Send Email Notification Rate-Limit Missing) Critical
ไฟล์ที่เกี่ยวข้อง: send-admin-email/index.ts

อาการ: หากมีผู้ใช้งานประสงค์ร้ายหรือเขียนสคริปต์สแปมยิง API การเติมเงิน ตัวระบบ Supabase Edge Function send-admin-email จะทำงานส่งอีเมลออกไปหาแอดมินทันทีไม่จำกัดจำนวนครั้ง ส่งผลให้อาจโดนระงับบัญชีอีเมล (เช่น Resend/Sendgrid) หรือมีค่าบริการส่วนเกินที่สูงมาก

สาเหตุ: ในระบบ Edge Function ไม่มีการควบคุมความถี่ในการเรียกใช้งาน (Rate Limiting) และไม่มีการตรวจสอบสิทธิ์ความปลอดภัยในระดับผู้ส่งที่รัดกุมพอ

แนวทางแก้ไข: เพิ่มระบบจำกัดความถี่ในการเรียกส่งข้อความด้วย Redis/Supabase Custom Cache หรือใช้การตรวจสอบ token ฝั่ง Client ร่วมกับสิทธิ์การส่ง

3. ประเด็นปัญหาระดับรุนแรงสูง (High Issues)

ประเด็นที่สร้างความสับสนแก่ผู้ใช้ หรือมีโอกาสทำให้ระบบการคิดคำนวณทำงานผิดพลาดได้บ่อยครั้ง:

ISSUE-4: การคำนวณจำนวนคนขับในพื้นที่ผิดพลาด (Driver Area Query Misleading) High
ไฟล์ที่เกี่ยวข้อง: parcel_service_screen.dart (ไลน์ 224-264)

อาการ: หน้าจอการจองรถส่งพัสดุของลูกค้าอาจแสดงจำนวนคนขับว่าง เช่น "พบคนขับว่าง 5 คนในบริเวณใกล้เคียง" แต่เมื่อกดส่งสินค้ากลับไม่มีคนขับคนใดรับงานได้เลย สร้างความสับสนและลดความพึงพอใจของลูกค้า

สาเหตุ: ฟังก์ชัน _checkNearbyOnlineDrivers ดึงรายชื่อคนขับออนไลน์ทุกคนมานับรวมโดยไม่ได้คัดกรองประเภทการให้บริการ (Service Type) ทำให้คนขับที่ให้บริการเฉพาะรับส่งอาหาร (Food) หรือรับส่งคน (Ride) ถูกนำมารวมเป็นคนขับส่งพัสดุด้วย

แนวทางแก้ไข: เพิ่มเงื่อนไขกรองประเภทการให้บริการในการสืบค้น เช่น กรองเฉพาะคนขับที่มีฟิลด์รองรับงานขนส่งพัสดุ service_type = 'parcel' เท่านั้น

ISSUE-5: การตั้งค่าล็อกอิน Supabase CLI ใน Sandbox ขัดข้อง (Supabase Deployment Blocked) High
ไฟล์ที่เกี่ยวข้อง: supabase/

อาการ: สคริปต์สำหรับการจัดการบัญชีและการกู้คืนระบบ (เช่น RPC migration ตัวล่าสุด) ไม่สามารถอัปโหลดหรือรันในสภาพแวดล้อม Production ได้จริง ทำงานได้เพียงระดับจำลองใน Local เท่านั้น

สาเหตุ: Supabase CLI ใน Sandbox ขาด Access Token หรือข้อมูลประจำตัวสำหรับการยืนยันสิทธิ์กับบริการคลาวด์

แนวทางแก้ไข: จำเป็นต้องรันคำสั่ง supabase login หรือระบุ SUPABASE_ACCESS_TOKEN ใน Environment Variables เพื่อปลดล็อกกระบวนการจัดส่งฐานข้อมูล

4. ประเด็นปัญหาปานกลาง (Medium Issues)

ประเด็นที่ส่งผลกระทบต่อความลื่นไหลของการใช้งาน (UX) หรืออาจก่อให้เกิดข้อมูลขยะตกค้างในระบบคลาวด์:

ISSUE-6: บันทึกใบจองพัสดุราคา ฿0 เมื่อการหาระยะทางขัดข้อง (Zero Price Booking Creation) Medium
ไฟล์ที่เกี่ยวข้อง: parcel_service_screen.dart (ไลน์ 382-386)

อาการ: มีการสร้างออเดอร์ส่งพัสดุราคา ฿0 บันทึกลงระบบฐานข้อมูล ซึ่งทำให้ระบบและคนขับขาดทุนในการให้บริการ

สาเหตุ: เมื่อเกิดความล้มเหลวในการคำนวณระยะทางจากพิกัด (เช่น พิกัดไม่มีในแผนที่) ระบบตั้งระยะทางทดแทนที่ 5.0 กิโลเมตร แต่กลับลืมเรียกฟังก์ชันคำนวณราคา _calculatePrice() ซ้ำ ส่งผลให้ตัวแปรราคายังเป็น 0.0 และถูกบันทึกเข้าระบบทันที

แนวทางแก้ไข: บังคับเรียกใช้งาน _calculatePrice() หลังการทำระยะทางทดแทน (Fallback) ทุกครั้ง และเพิ่มเงื่อนไขปฏิเสธการจองหากราคาคำนวณได้เป็น 0.0 หรือน้อยกว่า

ISSUE-7: ระบบติดตามสถานะแสดง "Unknown status" (Tracking Step Mismatch) Medium
ไฟล์ที่เกี่ยวข้อง: tracking_screen.dart (ไลน์ 531-574)

อาการ: ลูกค้าจะเห็นหน้าจออัปเดตความคืบหน้าขึ้นสถานะ "Unknown status" เป็นระยะๆ สร้างความกังวลว่าระบบเกิดข้อผิดพลาด

สาเหตุ: เมื่อคนขับเดินทางมาถึงจุดรับพัสดุ สถานะงานจะถูกตั้งเป็น arrived แต่ในหน้าจอติดตามสถานะฝั่งลูกค้าไม่มีการระบุข้อความเพื่อรองรับสถานะนี้ในคำสั่ง switch-case

แนวทางแก้ไข: เพิ่มกรณี case 'arrived': ในฟังก์ชัน _getStatusInfo() และ _getTimelineSteps() เพื่อให้แสดงข้อความที่เหมาะสม เช่น "คนขับรถเดินทางมาถึงจุดรับพัสดุเรียบร้อยแล้ว"

ISSUE-8: ไฟล์ภาพถ่ายพัสดุค้างในระบบจัดเก็บข้อมูล (Orphaned Storage Photos) Medium
ไฟล์ที่เกี่ยวข้อง: parcel_service_screen.dart (ไลน์ 373-408)

อาการ: มีไฟล์ภาพอัปโหลดค้างอยู่ใน Supabase Storage เป็นจำนวนมากโดยไม่มีข้อมูลออเดอร์เชื่อมโยง ทำให้สิ้นเปลืองพื้นที่จัดเก็บ

สาเหตุ: ระบบทำการอัปโหลดไฟล์ภาพพัสดุก่อนสร้างคำสั่งซื้อ หากการเชื่อมต่อสร้างคำสั่งซื้อในขั้นตอนต่อมาล้มเหลว ระบบจะข้ามการทำงานไปโดยไม่ได้ลบภาพที่อัปโหลดไปแล้วออก

แนวทางแก้ไข: เพิ่มคำสั่งลบรูปภาพใน Catch Block ของการทำงานส่งข้อมูล เพื่อให้ทำการ Rollback ลบไฟล์ภาพออกจากคลาวด์จัดเก็บทันทีหากการสร้างออเดอร์ล้มเหลว

ISSUE-9: หน้าจอแอดมินแจ้งเตือนซ้ำซ้อน 2 ชั้น (Double Confirmation Dialog in Admin) Medium
ไฟล์ที่เกี่ยวข้อง: ordersActions.js (ไลน์ 377)

อาการ: แอดมินต้องกดยืนยันการทำงานซ้ำซ้อน 2 ครั้ง และหน้าต่างยืนยันเบราว์เซอร์ (Native Dialog) ส่งผลให้เลย์เอาต์หน้าจอแอดมินค้างหากทำการกดยกเลิก

สาเหตุ: มีการเรียกใช้งานฟังก์ชันยืนยันดั้งเดิม confirm() ซ้ำซ้อนกับกล่องข้อความยืนยันที่เป็น HTML Modal สวยงามที่เพิ่งถูกปรับแต่งใหม่

แนวทางแก้ไข: ลบคำสั่ง confirm(confirmText) ของเบราว์เซอร์ดั้งเดิมออก และพึ่งพาเฉพาะการกดยืนยันผ่านระบบหน้าต่างลอยตัว (HTML Modal) เท่านั้น

5. ประเด็นปัญหาทั่วไป (Low Issues)

ข้อควรปรับปรุงเพื่อสุขอนามัยที่ดีของโค้ดเบส (Code Hygiene) และความสะดวกสบายในการใช้งาน:

ISSUE-10: โค้ดฟังก์ชันการอัปเดตรูปที่ไม่ได้ถูกเรียกใช้งาน (Dead Code in ParcelService) Low
ไฟล์ที่เกี่ยวข้อง: parcel_service.dart (ไลน์ 215)

อาการ: ฟังก์ชัน updateParcelPhoto() ถูกกำหนดไว้ในคลาสบริการแต่ไม่มีส่วนการทำงานใดเรียกใช้ ซึ่งหากมีผู้นำไปใช้ในอนาคต อาจเกิดช่องโหว่ให้สามารถเขียนทับภาพพัสดุหลังจากจัดส่งเสร็จสิ้นแล้วได้ เนื่องจากขาดการจำกัดสิทธิ์ในส่วนสถานะ

สาเหตุ: เขียนฟังก์ชันเตรียมไว้แต่ไม่ได้นำมาประกอบกับฝั่ง Client UI

แนวทางแก้ไข: ลบฟังก์ชันนี้ออกเพื่อความคลีน หรือเพิ่มเงื่อนไขความปลอดภัย parcel_status = 'created' กรองก่อนอัปเดต

ISSUE-11: ขนาดปุ่มกดปฏิกิริยาบนแผงแอดมินเล็กเกินไป (Touch Target Too Small) Low
ไฟล์ที่เกี่ยวข้อง: ordersActions.js

อาการ: แอดมินที่กดผ่านสมาร์ทโฟนหรือแท็บเล็ตมักจะกดยากหรือกดผิดไปโดนปุ่มข้างเคียง เนื่องจากพื้นที่ของปุ่มมีขนาดไม่ถึงเกณฑ์ขั้นต่ำ 44px (ตามหลักสากล UX/UI)

สาเหตุ: ขาดการกำหนดค่า CSS Padding และความสูงขั้นต่ำสำหรับหน้าจอมือถือโดยเฉพาะ

แนวทางแก้ไข: เพิ่มคลาส CSS ปรับความสูงขั้นต่ำและระยะขอบของปุ่มสำหรับ breakpoint ขนาดหน้าจอขนาดเล็ก
