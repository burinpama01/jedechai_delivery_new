Feature Plan v1 · 2026-06-09 · Laundry Pickup & Return

แผนเพิ่มฟีเจอร์ร้านซักผ้า บริการรับ-ส่งผ้า

เพิ่มบริการ ซักผ้า โดยลูกค้าเลือกได้ 2 แบบ: ให้คนขับมารับผ้าไปส่งร้านแล้วลูกค้าไปรับเองที่ร้านในราคาเที่ยวเดียว หรือให้คนขับมารับผ้าและส่งกลับหลังร้านซักเสร็จในราคาไปกลับ โดยขากลับต้องสร้างงานใหม่อัตโนมัติและไม่บังคับใช้คนขับเดิม

สรุปภาพรวม Requirement ทางเลือกออกแบบ Architecture ร้านซักผ้าแยก Food Chat Quote GP & ค่าส่ง Payment Flow UI & Dialogs Notify Data Model User Flow Implementation Tasks Verification Risks Rollout

1. สรุปภาพรวม

Recommended

แนวทางหลัก

ใช้ laundry_orders เป็นหัวออเดอร์ซักผ้า และใช้ bookings เป็นงานคนขับแต่ละขา โดยเพิ่ม service_type = laundry, laundry_leg = outbound|return, และ laundry_order_id เพื่อ reuse ระบบคนขับ, wallet, notification, tracking และรายงานเดิมให้มากที่สุด

Separate Storefront

ร้านซักผ้าไม่ใช่ร้านอาหาร

แยก flow ร้านซักผ้าออกจาก food merchant: ร้านซักผ้า CRUD laundry_packages เช่น ซักอบพับ, ซักแห้ง, ผ้านวม, รีดผ้า โดย package เป็นรายการบริการสำหรับขอประเมินราคาเท่านั้น ไม่ใช่ราคาจองทันที

Core Rule

ขากลับไม่ใช้ไรเดอร์เดิม

เมื่อร้านกด “ซักเสร็จ/พร้อมส่งกลับ” ระบบสร้าง return booking ใหม่โดยตั้ง driver_id = null และ status = pending จากพิกัดร้านไปพิกัดลูกค้า จากนั้นแจ้งคนขับใกล้ร้านผ่าน flow งานใหม่ตามปกติ

Customer Discovery

แสดงร้านใกล้สุดก่อน

หน้า customer laundry ต้องเรียงร้านซักผ้าตามระยะจากจุดรับของลูกค้า และแสดงค่าส่ง estimate ต่อร้านทันที เพื่อให้ลูกค้าเห็น “ร้านใกล้ + ค่าส่งถูก” ก่อนตัดสินใจ

เหตุผล: โครงสร้างปัจจุบันมี Booking, BookingStatus, DriverJobVisibilityPolicy, get_nearby_bookings, wallet และ notification อยู่แล้ว จึงควรให้ laundry ใช้ quote/order layer แยก แล้วสร้าง booking เฉพาะหลังลูกค้ายืนยัน quote และชำระเงินตาม rule แล้ว

2. Requirement ที่ล็อกจากโจทย์

  • ลูกค้าเลือก ร้านซักผ้า และเลือกบริการรับผ้า
  • ร้านซักผ้าต้องมี flow แยกจากร้านอาหาร และไม่ใช้รายการอาหาร/cart/menu option เดิม
  • ร้านซักผ้า CRUD รายการ package ของร้านได้ เช่น ซักอบพับ, ซักแห้ง, ผ้านวม, รีดผ้า, บริการด่วน
  • Package เป็นแม่แบบขอประเมินราคาเท่านั้น ไม่มีราคาเหมาจองทันที และลูกค้าไม่สามารถกดบุ๊คทันทีจากราคาแพ็กเกจ
  • ทุกออเดอร์ต้องเริ่มจาก ขอราคา / ส่งคำขอให้ร้านประเมิน เพื่อให้ร้านยืนยันค่าซักจริงก่อนสร้าง booking
  • การขอราคาเป็นรูปแบบ chat กับร้าน แต่ต้องแสดงรายละเอียดคำขอครบถ้วนเป็น structured summary
  • Form ขอราคา/ส่งคำขอให้ร้านประเมินต้องแนบหลักฐานภาพ เช่น รูปถุงผ้า, รูปผ้า, รูปคราบ, หรือรูปจำนวนโดยประมาณ
  • ลูกค้าเข้าฟีเจอร์ร้านซักผ้าแล้วระบบต้องแสดงร้านใกล้สุดก่อน พร้อมราคาค่าส่งตามระยะทางของแต่ละร้าน
  • ต้องระบุรูปแบบการคิด GP และค่าส่ง แยกชัดระหว่างค่าซักของร้าน, GP ระบบ, ค่าส่งขาไป, ค่าส่งขากลับ และรายได้คนขับ
  • Driver จ่ายร้านเป็นยอดหลังหัก GP เท่านั้น: merchant_net_amount = laundry_amount - platform_gp_amount
  • หลังแอดมินอนุมัติร้านใหม่ ต้องมี flow ให้ร้านเลือกประเภทร้าน เช่น ร้านอาหาร หรือ ร้านซักผ้า เพื่อแยก storefront/schema/workflow ตั้งแต่ onboarding
  • Quote ต้องมีเวลาหมดอายุ ถ้าลูกค้าไม่ตอบรับ/ไม่กดยืนยันภายในเวลา ระบบปิดออเดอร์อัตโนมัติเป็น booking ล้มเหลวฝั่งร้าน-ลูกค้า โดยยังไม่สร้าง outbound booking ไปหาคนขับ
  • ค่าเริ่มต้น quote_expires_at คือ 1 ชั่วโมงหลังร้านส่ง quote และร้านค้าสามารถตั้งค่าเวลาหมดอายุ quote ได้เองใน settings
  • ร้านซักผ้าต้องแยก quote และ outbound booking ชัดเจน: quote เป็นงานระหว่างร้านกับลูกค้า; outbound booking เป็นงานคนขับที่สร้างหลัง quote accepted + payment guard เท่านั้น
  • ต้องมี notification ที่จำเป็นทุกจุด แต่ไม่ spam: notify เมื่อมี action ที่ต้องให้ role ถัดไปตัดสินใจหรือรับทราบสถานะสำคัญ
  • ร้านต้องได้ notify พร้อมเสียงเตือนเมื่อมี quote request ใหม่ เพื่อไม่พลาดคำขอประเมินราคา
  • เมื่อสร้าง outbound booking แล้วไม่มีคนขับกดรับภายในเวลาที่กำหนด แอดมินต้องจัดการโยน/assign งานให้คนขับได้แบบเดียวกับ food
  • ร้านซักผ้าใช้ บัญชี merchant เดิม ไม่สร้าง role ใหม่; หลังอัปเดตระบบต้องเช็คก่อนว่า merchant มี merchant_service_types แล้วหรือยัง ถ้ายังไม่มีให้บังคับเปิดหน้าเลือกประเภทร้านก่อนเข้า dashboard
  • MVP บังคับคนขับถ่ายภาพหน้างานเป็นหลักฐานก่อนกด “ยืนยันรับผ้า” ทุกครั้ง
  • Wallet hold/release ใน MVP ให้สรุป contract ของ RPC เดิมก่อน implementation และ reuse RPC/ledger flow เดิมเท่าที่รองรับก่อนเพิ่ม ledger type ใหม่
  • ลูกค้าต้องจ่าย ค่าซักที่ร้าน quote + ค่าส่งขาไป ก่อนเริ่มงานขาไป
  • ถ้าลูกค้าอยู่ที่ทำงานแต่ให้รับผ้าที่บ้าน หรือไม่ได้อยู่ที่จุดรับ ระบบต้องบังคับจ่ายด้วย wallet เท่านั้น
  • ถ้าลูกค้าอยู่ที่บ้าน/จุดรับตามหมุด ลูกค้าเลือกจ่ายได้ cash หรือ wallet
  • ถ้าจ่ายด้วย wallet เมื่อคนขับกดยืนยันรับผ้าพร้อมหลักฐาน เงินจะเข้า wallet คนขับ แล้วคนขับจ่ายค่าซักให้ร้านเองตอนส่งผ้าถึงร้าน
  • ถ้าเลือกขากลับ ลูกค้าต้องเลือกวิธีชำระค่าส่งขากลับแยกเป็น cash หรือ wallet; คนขับขากลับเก็บเฉพาะ ค่าส่งขากลับ ไม่เก็บค่าซักซ้ำ
  • ขากลับแบบ cash ใช้ flow ปกติ คนขับเก็บเงินลูกค้าที่จุดหมายเมื่อส่งผ้าเสร็จ
  • ขากลับแบบ wallet ต้องหัก/hold wallet ลูกค้าตั้งแต่เลือกส่งกลับถึงบ้าน โดยลูกค้าต้องมียอดเงินเพียงพอก่อนจึงเลือกได้ และ release เข้า wallet คนขับขากลับเมื่อส่งงานเสร็จพร้อมแจ้งเตือนให้ตรวจสอบ
  • ตัวเลือก A: คนขับมารับผ้าแล้วลูกค้าไปรับเองที่ร้าน คิดราคาเที่ยวเดียว
  • ตัวเลือก B: คนขับมารับผ้าและส่งกลับหลังซักเสร็จ คิดราคาไปกลับ
  • เมื่อร้านรับออเดอร์ซักเสร็จและกดพร้อมส่งกลับ ระบบต้องสร้างงานขากลับใหม่อัตโนมัติ
  • ขากลับไม่จำเป็นต้องใช้ไรเดอร์เดิม คนขับคนใดก็ได้ที่เข้าเงื่อนไขรับงานได้
  • ต้องรักษา semantics เงินเดิม: non-food ใช้ bookings.price เป็นค่าขนส่งของงานนั้น ไม่เอายอดรวมหลายขาไปปนใน booking เดียว

3. ทางเลือกออกแบบ

ทางเลือก แนวคิด ข้อดี ข้อเสีย ผลตัดสิน
A. Booking เดียวทั้งงาน ใช้ booking เดิม แล้ว reset driver_id สำหรับขากลับ แก้น้อยตอนแรก เสี่ยงพัง history, earnings, assigned_at, completed_at, notification และ audit trail ไม่แนะนำ
B. ระบบซักผ้าแยกบางส่วน แยก laundry quote/catalog/chat/order ออกจาก food แต่ reuse bookings, wallet, notification, driver dispatch และ admin reporting เฉพาะส่วนที่เหมาะสม แยก domain ซักผ้าชัดเจน, ไม่ปน food menu/cart, quote ผ่าน chat ได้, reuse infrastructure เดิม, ขากลับเป็นงานใหม่จริง ต้องเพิ่ม migration/RLS/RPC ใหม่ และต้องออกแบบ contract ระหว่าง laundry order กับ booking legs ให้แน่น แนะนำ
C. ระบบซักผ้าแยกทั้งหมด สร้าง dispatch/payment/status ใหม่สำหรับ laundry ยืดหยุ่นที่สุดระยะยาว ซ้ำกับระบบเดิมมาก, ใช้เวลานาน, regression risk สูง เก็บไว้ Phase 2+

4. Architecture ที่เสนอ

Customer เลือกบริการซักผ้า, ร้าน, จุดรับ, mode รับเอง/ส่งกลับ
Laundry Order เก็บสถานะงานซัก, mode, quote, booking ids
Outbound Booking laundry/outbound ลูกค้า → ร้าน
Merchant Ready ร้านกดซักเสร็จ; self-pickup จบที่ notify, return mode สร้างงานใหม่
Return Booking laundry/return ร้าน → ลูกค้า, driver_id = null

Boundaries

  • laundry_orders: source of truth ของธุรกิจซักผ้า เช่น mode, สถานะซัก, quote รวม, booking ขาไป/กลับ
  • bookings: source of truth ของงานคนขับแต่ละเที่ยว เช่น status, driver, distance, price, earnings
  • mark_laundry_ready_guarded: RPC atomic สำหรับร้านกดพร้อมรับเองหรือพร้อมส่งกลับ และป้องกันการสร้าง return booking ซ้ำ
  • confirm_laundry_quote_guarded: RPC atomic สำหรับลูกค้ายืนยัน quote, ตรวจ payment rule และสร้าง outbound booking หลังจ่าย/เลือกวิธีจ่ายถูกต้องแล้ว
  • confirm_laundry_pickup_guarded: RPC atomic สำหรับคนขับยืนยันรับผ้าพร้อมหลักฐาน และ release wallet ให้คนขับเมื่อ flow เป็น wallet
  • DriverJobVisibilityPolicy และ get_nearby_bookings: เพิ่มกฎให้ laundry มองเห็นเหมือน parcel เมื่อ status เป็น pending

4.1 Flow ร้านซักผ้าแยกจาก Food

Not Food

สิ่งที่ห้าม reuse ตรง ๆ

  • ไม่ใช้ menu_items, booking_items แบบอาหาร, cart provider ของอาหาร หรือ option group อาหารเป็น UI หลัก
  • ไม่ใช้คำว่าอาหาร เช่น “เตรียมอาหาร”, “อาหารพร้อมรับ”, “ค่าอาหาร” ใน laundry status/detail
  • ไม่รวม laundry order เข้า tab food order โดยไม่มี filter แยก เพราะ merchant workflow คนละแบบ
Laundry Catalog

รายการร้านซักผ้า

  • laundry_packages: รายการบริการของร้าน เช่น ซักอบพับ, ซักแห้งสูท, ผ้านวม 6 ฟุต
  • Package ใช้เป็นแม่แบบให้ลูกค้าส่งคำขอประเมินราคา ไม่ใช่ราคาเหมาจองทันที
  • รองรับราคาอ้างอิง/ช่วงราคา, หน่วยประเมิน, ระยะเวลาทำงาน, service tag และสถานะเปิด/ปิดรายการ
Merchant Console

หน้าร้านซักผ้า

  • เพิ่ม console หรือ tab แยกชื่อ “Laundry” ใน merchant app
  • CRUD package, เวลาให้บริการ, pickup capacity และ order status เฉพาะซักผ้า
  • รับคำขอประเมิน, ตรวจรายการ/รูป/จำนวน, ส่ง quote, ยืนยันรับผ้า, ยืนยันรับเงินจากคนขับ, เริ่มซัก, ซักเสร็จ, ส่งกลับ/รอลูกค้ามารับ
Customer Discovery

ค้นหาร้านใกล้สุด

  • ใช้พิกัดจุดรับของลูกค้าเป็น origin สำหรับ query ร้านซักผ้า
  • เรียงตามระยะทางน้อยไปมาก และคำนวณ outbound delivery fee ต่อร้าน
  • ถ้าเลือกส่งกลับ ให้คำนวณ return delivery fee จากร้านกลับปลายทางลูกค้าและแสดงราคารวมไปกลับ

4.2 Chat Quote และ Form ขอราคา

Structured Chat

ขอราคาเป็น chat กับร้าน

  • ลูกค้าส่งคำขอจาก form แล้วระบบเปิด thread chat ระหว่างลูกค้าและร้าน
  • บนหัว chat ต้องมี structured summary: package, จุดรับ, return mode, pickup presence, รูปแนบ, note, ค่าส่ง estimate
  • ร้านตอบ/ถามเพิ่มใน chat ได้ แต่การส่ง quote ต้องใช้ action ส่งใบเสนอราคา ที่เก็บเป็น structured data
Evidence Required

Form ต้องแนบรูป

  • บังคับแนบอย่างน้อย 1 รูปก่อนส่งคำขอ เช่น รูปถุงผ้า/รูปกองผ้า/รูปคราบ/รูปป้ายดูแลผ้า
  • รองรับหลายรูป พร้อมคำอธิบายสั้น ๆ ต่อรูปถ้ามี
  • รูปแนบต้องผูกกับ laundry_quote_requests และแสดงให้ร้าน/แอดมินเห็นใน audit
Merchant Quote

ร้านส่ง quote จาก chat

  • ร้านกรอกค่าซักจริง, note, เวลาประมาณเสร็จ, เงื่อนไขพิเศษ และวันหมดอายุ quote
  • ระบบคำนวณค่าส่งจากพิกัด ไม่ให้ร้านแก้เอง ยกเว้น admin override
  • เมื่อส่ง quote ต้อง notify ลูกค้าให้ยืนยันและเลือก payment rule
No Booking Yet

ยังไม่สร้าง booking

  • Chat/quote request อยู่ใน laundry domain เท่านั้น
  • Outbound booking สร้างหลังลูกค้ายืนยัน quote และผ่าน payment guard แล้ว
  • ถ้าลูกค้าปฏิเสธหรือ quote หมดอายุ ออเดอร์จบโดยไม่มี driver job

4.3 รูปแบบการคิด GP และค่าส่ง

ยอดเงินใครเป็นเจ้าของวิธีคิดหมายเหตุ
laundry_amount ร้านซักผ้า ร้านส่ง quote จาก chat หลังตรวจรูป/รายละเอียด เป็นฐานคิด GP ระบบของร้าน
platform_gp_amount ระบบ laundry_amount * laundry_gp_rate หรือ rate เฉพาะร้าน ถ้ามี custom config หักจากรายได้ร้าน ไม่หักจาก driver
merchant_net_amount ร้านซักผ้า laundry_amount - platform_gp_amount Driver จ่ายร้านยอดนี้เท่านั้น เพื่อให้ GP ระบบถูกหักก่อนถึงร้าน
outbound_delivery_fee คนขับขาไป สูตร service rate ของ laundry: base + per km จากจุดรับลูกค้าไป ร้าน ลูกค้าจ่ายพร้อมค่าซักก่อนสร้าง booking ขาไป
return_delivery_fee คนขับขากลับ สูตร service rate ของ laundry: base + per km จากร้านไปปลายทางลูกค้า เก็บเมื่อมี return booking เท่านั้น และไม่รวมค่าซักซ้ำ
Decision ล็อกแล้ว: ถ้า wallet flow ให้ driver จ่ายร้านเป็นยอด merchant_net_amount หลังหัก GP แล้วเท่านั้น ระบบต้องทำ ledger แยก platform_gp_amount เพื่อ audit ได้

4.4 Payment Flow ที่ QA แล้ว

Quote Only

ไม่มีราคาเหมาจองทันที

  • ลูกค้าเลือก package เพื่อส่งคำขอให้ร้านประเมินเท่านั้น
  • ร้านต้องส่ง quote ที่ประกอบด้วย laundry_amount + outbound_delivery_fee
  • Quote มีเวลาหมดอายุ ถ้าลูกค้าไม่ยืนยันภายในเวลา ให้ auto-close เป็น quote_expired/failed โดยไม่สร้าง outbound booking
  • ระบบสร้าง outbound booking หลังลูกค้ายืนยัน quote และผ่าน payment rule แล้วเท่านั้น
Pickup Payment

จ่ายก่อนเริ่มขาไป

  • ยอดที่ต้องจ่ายก่อนเริ่มงาน = ค่าซักตาม quote + ค่าส่งขาไป
  • ลูกค้าอยู่จุดรับตามหมุด: เลือก cash หรือ wallet
  • ลูกค้าไม่ได้อยู่จุดรับ เช่น อยู่ที่ทำงานแต่ให้รับที่บ้าน: ใช้ wallet เท่านั้น
Wallet Handoff

Wallet เข้า driver ก่อนจ่ายร้าน

  • เมื่อจ่าย wallet ระบบ hold/authorize ยอดไว้หลังลูกค้ายืนยัน quote โดย MVP ต้องสรุปและ reuse RPC เดิมก่อนเพิ่ม ledger type ใหม่
  • เมื่อคนขับกด “ยืนยันรับผ้าแล้ว” พร้อมภาพถ่ายหน้างาน ระบบโอนยอดค่าซัก + ค่าส่งขาไปเข้า wallet คนขับ
  • คนขับจ่ายค่าซักให้ร้านเองตอนส่งผ้าถึงร้าน และร้านต้องกดยืนยันรับเงินก่อนเริ่มซัก
Return Leg

ขากลับเก็บเฉพาะค่าส่ง

  • ถ้าลูกค้าเลือกส่งกลับ ร้านกดซักเสร็จแล้วระบบสร้าง return booking ใหม่
  • ตอนเลือก return_delivery ลูกค้าต้องเลือกวิธีชำระค่าส่งขากลับ: cash หรือ wallet
  • ถ้าเลือก cash คนขับขากลับเก็บเงินลูกค้าที่จุดหมายตาม flow ปกติ
  • ถ้าเลือก wallet ระบบต้องตรวจยอด wallet ให้เพียงพอและ hold ค่าส่งขากลับตั้งแต่ตอนเลือก mode ส่งกลับถึงบ้าน
  • เมื่อคนขับขากลับส่งงานเสร็จ ระบบ release ยอด hold เข้า wallet คนขับขากลับ และแจ้งเตือนลูกค้า/คนขับให้ตรวจสอบ
  • ห้ามเก็บค่าซักซ้ำในขากลับ และห้ามรวมรายได้ขาไป/ขากลับเป็น booking เดียว

  • หลังแอดมินอนุมัติ merchant account แล้ว ร้านต้องเข้า flow เลือกประเภทร้านก่อนเปิดใช้งานจริง
  • หลังอัปเดตแอป/เว็บหรือ login ใหม่ ต้องเช็ค profiles.merchant_service_types; ถ้ายังเป็น null/empty ให้บังคับเปิดหน้าเลือกประเภทร้านและห้ามเข้า dashboard จนเลือกเสร็จ
  • ตัวเลือกเริ่มต้น: food = ร้านอาหาร, laundry = ร้านซักผ้า
  • การเลือกประเภทร้านต้องกำหนด profiles.merchant_service_types, route หน้า dashboard, catalog schema และ setup checklist ของร้าน
  • ร้านอาหารไป flow menu/food order เดิม; ร้านซักผ้าไป flow package/quote/chat/laundry order
  • ถ้าต้องเปลี่ยนประเภทร้านภายหลัง ให้ผ่าน admin action พร้อม audit log เท่านั้น

6. User Flow

Customer

สร้างออเดอร์

  1. เปิดหน้า “ซักผ้า” จาก customer home
  2. เลือกจุดรับผ้าเพื่อใช้ค้นหาร้านซักผ้าใกล้ที่สุด
  3. ระบบแสดงร้านซักผ้าเรียงจากใกล้สุด พร้อม outbound delivery fee estimate และราคาอ้างอิงของ package
  4. เลือกร้านซักผ้า แล้วเลือก package ของร้านนั้น ไม่ใช้เมนูอาหาร
  5. ส่งคำขอประเมินราคา พร้อมรายละเอียด/รูป/จำนวนโดยประมาณ และยังไม่สร้าง booking
  6. ร้านส่ง quote กลับมาเป็นค่าซักจริง + ค่าส่งขาไป พร้อมเวลาหมดอายุค่าเริ่มต้น 1 ชั่วโมง หรือตามค่าที่ร้านตั้งไว้
  7. ระบุที่อยู่ปลายทางสำหรับส่งกลับเมื่อเลือก mode ส่งกลับถึงบ้าน
  8. เลือก mode: รับเองที่ร้าน หรือ ส่งกลับถึงบ้าน; ถ้าเลือกส่งกลับต้องเลือกวิธีจ่ายค่าส่งขากลับเป็น cash/wallet
  9. ถ้าเลือกขากลับแบบ wallet ระบบต้องตรวจยอดเงินและ hold ค่าส่งขากลับทันที ถ้า wallet ไม่เพียงพอให้เลือกไม่ได้
  10. เลือกว่าลูกค้าอยู่ที่จุดรับหรือเป็น remote pickup; remote pickup ต้องจ่าย wallet เท่านั้น
  11. ยืนยัน quote และจ่ายค่าซัก + ค่าส่งขาไปก่อนหมดอายุ ระบบจึงสร้าง outbound booking
  12. ถ้าลูกค้าไม่ยืนยันก่อนหมดอายุ ระบบปิดเป็น booking ล้มเหลวฝั่งร้าน-ลูกค้า และไม่แจ้งงานให้คนขับ
Driver

งานขาไป/ขากลับ

  1. เห็นงาน laundry เมื่อ online, อยู่ใน radius, และรับ service type นี้
  2. รับงานขาไปจากลูกค้า/จุดรับไปส่งร้าน
  3. ก่อนรับงานเห็นชัดว่าจ่าย cash หรือ wallet และต้องจ่ายร้านเท่าไร
  4. เมื่อรับผ้าต้องถ่ายภาพหน้างานก่อนกดยืนยัน; ถ้า wallet เงินค่าซัก + ค่าส่งขาไปเข้า wallet คนขับ
  5. เมื่อส่งถึงร้าน คนขับจ่ายค่าซักให้ร้าน แล้วร้านกดยืนยันรับเงิน/รับผ้า
  6. งานขากลับเป็น booking ใหม่ คนขับเดิมหรือคนอื่นก็รับได้; ถ้า return wallet ให้เติมเข้า wallet คนขับขากลับหลังส่งงานเสร็จ

  5. กดยืนยันรับผ้าและรับเงินจากคนขับก่อนเริ่มซัก
  6. กด “ซักเสร็จ”
  7. ถ้า self_pickup: แจ้งลูกค้าให้ไปรับเอง
  8. ถ้า return_delivery: เรียก RPC สร้าง return booking อัตโนมัติตามวิธีจ่ายขากลับที่ลูกค้าเลือกไว้

  • เงินซ้ำหรือรายได้คนขับผิด: อย่าเก็บราคาไปกลับทั้งก้อนใน booking ขาไป; ให้ booking แต่ละขาเก็บ fare ของขานั้น
  • book ก่อน quote: ระบบต้องไม่สร้าง booking จนกว่าร้านส่ง quote และลูกค้ายืนยัน quote/payment แล้ว
  • quote ค้างร้าน: quote ต้องหมดอายุอัตโนมัติและย้ายเป็น failed ถ้าลูกค้าไม่ยืนยันภายในเวลา
  • quote expiry config drift: quote ใหม่ใช้ setting ร้านล่าสุด แต่ quote ที่ส่งแล้วต้องยึด quote_expires_at ที่บันทึกไว้
  • ร้านพลาด quote ใหม่: merchant app ต้องมี notify พร้อมเสียง แต่ต้องกันเสียงซ้ำจาก realtime reconnect
  • สับสน booking สองแบบ: UI/admin ต้องแยก `laundry quote/order` ออกจาก `outbound booking` ชัดเจน เพราะ quote ยังไม่ใช่งานคนขับ
  • outbound ไม่มี driver: ถ้าลูกค้าจ่ายแล้วแต่ไม่มีคนขับรับ ต้อง escalate เข้า admin queue เพื่อโยนงานเหมือน food
  • merchant type ผิด: ร้านซักผ้าห้ามเข้าหน้า food menu/order โดยค่าเริ่มต้น และร้านอาหารห้ามเห็น laundry package/quote flow เว้นแต่ admin เพิ่ม service type
  • merchant เดิมไม่มี type หลังอัปเดต: ต้อง block ที่หน้าเลือกประเภทร้าน ไม่ default เป็น food เงียบ ๆ เพราะอาจซ่อน flow laundry หรือเปิด flow ผิดร้าน
  • form ไม่มีรูป: ต้อง block การส่งคำขอถ้าไม่มีรูปหลักฐานอย่างน้อย 1 รูป
  • chat ขาด structured data: chat ใช้คุยได้ แต่ quote/payment ต้องมาจาก action structured เท่านั้น
  • GP drift: ต้องคำนวณ GP ฝั่ง server จาก quote/config ไม่เชื่อค่าจาก client
  • notify spam: notify เฉพาะ event ที่ต้อง action หรือเปลี่ยนสถานะสำคัญ ไม่ยิงซ้ำทุก polling/realtime update
  • remote pickup ใช้ cash: ถ้าลูกค้าไม่ได้อยู่จุดรับ ห้ามเลือก cash ต้อง wallet เท่านั้น
  • wallet release เท็จ: MVP ต้องบังคับภาพถ่ายหน้างานก่อนยืนยันรับผ้าและก่อนเงินเข้า wallet คนขับ
  • เพิ่ม ledger type ใหม่เร็วเกินไป: ต้องสรุป RPC/ledger เดิมก่อน ถ้า RPC เดิมรองรับได้ให้ reuse ก่อนเพื่อลด regression wallet
  • return wallet เงินไม่พอ: ต้อง block การเลือกส่งกลับแบบ wallet ตั้งแต่ตอนเลือก mode ไม่รอไป fail ตอนสร้าง return booking
  • return wallet release ซ้ำ: ต้องมี idempotency จาก hold transaction/status เพื่อ release ค่าส่งขากลับเข้า wallet คนขับครั้งเดียว
  • return cash/wallet สับสน: job card ขากลับต้องแสดงชัดว่าเก็บเงินสดที่ปลายทางหรือรอ wallet release หลังส่งงานเสร็จ
  • ร้านยังไม่ได้รับเงิน: ร้านต้องกดยืนยันรับเงินจากคนขับก่อนเปลี่ยนเป็นเริ่มซัก
  • สร้างขากลับซ้ำ: RPC ต้อง lock row และเช็ค return_booking_id ก่อน insert
  • RLS leak: ร้านซักผ้าต้องเห็นเฉพาะ order ของตัวเอง; driver เห็นเฉพาะงาน visible/assigned
  • status drift: อย่า reuse food status text ตรง ๆ จน UX สับสน เช่น “อาหารพร้อมรับ” ต้องเป็น “ผ้าพร้อมรับ”
  • food schema ปน laundry: อย่า reuse menu/cart อาหารเป็น data model หลักของร้านซักผ้า เพราะ package quote และ workflow ต่างกัน
  • คนขับเดิมถูกบังคับโดยไม่ตั้งใจ: return booking ต้องไม่ copy driver_id จาก outbound booking
  • ร้านหลายประเภท: ต้องมี field service type ของ merchant เพื่อไม่ให้ร้านอาหารเดิมโผล่ในหน้าเลือกซักผ้าโดยไม่ตั้งใจ

10. Rollout / Done Criteria

  1. เพิ่ม feature flag เช่น system_config.laundry_service_enabled เพื่อซ่อนเมนูจนพร้อม
  2. ทำ migration ใน dev/staging ก่อน production และเตรียม rollback SQL สำหรับตาราง/คอลัมน์ใหม่
  3. ก่อน build ต้อง bump version ใน jedechai_delivery_new/pubspec.yaml ตามกฎ repo
  4. หลัง implementation ต้องเรียก code_reviewer ตรวจ diff เพราะกระทบ DB, payment/earnings, RLS, notification และ runtime flow
  5. ห้าม deploy production จนกว่าจะยืนยัน Supabase project, backup, migration result, และ secrets ที่เกี่ยวข้อง
Scope ที่ล็อกแล้วก่อนลง implementation: ร้านซักผ้าใช้บัญชี merchant เดิมและบังคับเลือก merchant_service_types เมื่อยังไม่มี type; MVP บังคับถ่ายภาพหน้างานก่อนยืนยันรับผ้า; wallet hold/release ต้องสรุปและ reuse RPC เดิมก่อนเพิ่ม ledger type ใหม่; ขากลับให้ลูกค้าเลือก cash/wallet โดย cash เก็บที่ปลายทาง ส่วน wallet ต้อง hold ตั้งแต่เลือกส่งกลับถึงบ้านและ release เข้า wallet คนขับขากลับเมื่อส่งงานเสร็จ
