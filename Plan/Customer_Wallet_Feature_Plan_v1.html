เพิ่มกระเป๋าเงินสำหรับลูกค้าเพื่อเติมเงิน, จ่ายค่าออเดอร์, รับ refund และดูประวัติธุรกรรม โดยใช้โครงสร้าง wallet เดิมของ Supabase เป็นฐาน แต่เพิ่ม guardrail สำหรับยอดเงินจริงให้ครบก่อน implementation.

Recommendation

ทำแบบ Hybrid Ledger: ใช้ตาราง wallets, wallet_transactions, topup_requests ที่มีอยู่แล้ว และเพิ่ม RPC เฉพาะ customer payment/refund เพื่อให้ booking + wallet debit/credit เป็น atomic มากที่สุด.

เหตุผล: ลด schema ใหม่, ใช้ admin approval/topup flow เดิมได้, แต่ยังปิด race condition และยอด drift จาก payment/coupon ได้.

1. Executive Summary

Build

ลูกค้าเติมเงิน

เพิ่มหน้า customer wallet, เติมเงินผ่าน PromptPay/manual request ก่อน และ reuse admin approval flow ของ topup_requests.

Build

จ่ายด้วย Wallet

เพิ่ม payment_method = wallet ให้ checkout หลัก โดย deduct หลังคำนวณยอดสุทธิและก่อนยืนยัน booking สำเร็จ.

Guardrail

Refund / Cancel

คืนเงินเข้ากระเป๋าด้วย transaction type ชัดเจน เมื่อออเดอร์ถูกยกเลิกในช่วงที่เข้าเงื่อนไข refund.

2. Requirements

Functional

  • ลูกค้าดูยอดคงเหลือ wallet ได้จากหน้า profile/account และ checkout.
  • ลูกค้าสร้างคำขอเติมเงินได้ โดยมี min/max amount และประวัติคำขอ.
  • แอดมินอนุมัติ/ปฏิเสธ top-up ผ่านระบบเดิมได้ โดยต้องรองรับ role customer ไม่ใช่เฉพาะ driver.
  • ลูกค้าเลือกจ่ายด้วย wallet ใน food, ride, parcel และ service ที่ใช้ booking ได้.
  • ระบบบันทึก transaction ทุกครั้ง: topup, payment, refund, adjustment.
  • ลูกค้าเห็น transaction history ของตัวเองเท่านั้น.

Non-functional

  • ยอดเงินต้องไม่ติดลบสำหรับ customer wallet.
  • หักเงินต้อง atomic และ idempotent ต่อ booking หนึ่งรายการ.
  • คำนวณยอดที่ต้องหักจาก source of truth เดียวกับ checkout: food = subtotal + delivery fee - coupon discount.
  • ห้ามให้ client update wallets.balance ตรงสำหรับ payment/refund.
  • RLS ต้องกันลูกค้าเห็น wallet/transaction ของคนอื่น.
  • ใช้ timezone policy เดิมผ่าน AppTime เมื่อแสดงเวลาในแอป.

4. Proposed Architecture

Backend Boundary

เพิ่ม RPC ฝั่ง Supabase สำหรับ customer wallet โดยไม่ให้ Flutter update balance ตรง:

  • customer_wallet_pay_booking(p_booking_id uuid, p_expected_amount numeric)
  • customer_wallet_refund_booking(p_booking_id uuid, p_reason text)
  • ensure_wallet_for_user(p_user_id uuid) หรือปรับ wallet_topup ให้ create wallet เมื่อไม่มี

Flutter Boundary

แยก service ใช้งานทั่วไปออกจาก driver-only wording:

  • คง WalletService แต่เพิ่ม method generic: getWallet, getTransactions, requestTopup, payBookingWithWallet.
  • สร้างหน้า customer wallet ใหม่ใน lib/apps/customer/screens/.
  • เพิ่ม wallet payment option ใน checkout/service screens โดยใช้ยอด final total ที่คำนวณแล้ว.

5. Data Model / Migration

Reuse Existing Tables

Table ใช้ทำอะไร Change ที่ต้องตรวจ/เพิ่ม
wallets เก็บยอดคงเหลือต่อ user ยืนยัน unique constraint user_id, เพิ่ม index ถ้ายังไม่มี, policy ให้ user select own wallet
wallet_transactions ledger ธุรกรรม topup/payment/refund เพิ่ม/ยืนยัน related_booking_id, type, description, created_at, RLS select own ผ่าน wallet
topup_requests คำขอเติมเงินที่รอแอดมินอนุมัติ เพิ่ม role_snapshot หรือรองรับ customer ใน query/admin UI, เพิ่มหลักฐานสลิปถ้าต้องใช้
bookings ออเดอร์หลักของระบบ ยืนยัน/เพิ่ม payment_method, payment_status, wallet_transaction_id, paid_at

Recommended Migration

สร้าง migration ใหม่ supabase/migrations/YYYYMMDDHHMMSS_customer_wallet_payment.sql เพื่อทำเฉพาะส่วนที่ยังขาด:

  • constraint กัน wallet ซ้ำต่อ user: unique(user_id).
  • function สำหรับจ่าย booking ด้วย wallet แบบ atomic.
  • function สำหรับ refund booking แบบ idempotent.
  • policy/RLS เพิ่มเติมถ้า policy เดิมยัง driver-centric.
  • index: wallet_transactions(wallet_id, created_at desc), wallet_transactions(related_booking_id).

6. User Flow

Top-up Flow

  1. ลูกค้าเปิดหน้า Wallet.
  2. ระบบแสดง balance, pending topups, recent transactions.
  3. ลูกค้าเลือกจำนวนเงินหรือกรอกเองตาม min/max.
  4. ระบบสร้าง topup_requests พร้อม status pending.
  5. แอดมินอนุมัติใน admin-web/admin app.
  6. Edge Function เรียก wallet_topup และบันทึก transaction type topup.

Payment Flow

  1. ลูกค้าเลือก wallet ใน checkout.
  2. UI เช็ก balance เทียบ final total เฉพาะเพื่อ UX เท่านั้น.
  3. เมื่อกดยืนยัน ระบบสร้าง booking เป็น pending payment หรือใช้ RPC create+pay ถ้าปรับเพิ่มได้.
  4. RPC หัก wallet ด้วย FOR UPDATE, ตรวจยอดไม่ติดลบ, บันทึก transaction.
  5. booking ถูก mark เป็น paid พร้อม wallet_transaction_id.
  6. ถ้า coupon usage fail ต้อง rollback/cancel booking และ refund wallet ทันที.

7. Implementation Tasks

Task 1: Database audit + migration

  • ตรวจ schema จริงของ wallets, wallet_transactions, topup_requests, bookings.
  • เพิ่ม migration เฉพาะ customer wallet payment/refund.
  • เพิ่ม SQL tests หรือ smoke SQL สำหรับ insufficient balance, duplicate payment, refund duplicate.

Task 2: Edge Function / Admin top-up

  • ปรับ supabase/functions/admin-actions/index.ts ให้ approve topup รองรับ customer wallet.
  • เปลี่ยน read-then-write wallet update เป็น RPC ถ้ายังพบใน approve/reject path.
  • เพิ่ม admin notification payload type แยก customer top-up.

Task 3: Flutter common wallet service

  • ปรับ WalletService ให้ generic ไม่ผูกชื่อ DriverWallet ใน API ใหม่.
  • เพิ่ม model Wallet / WalletTransaction ถ้ายังไม่มี generic model.
  • เพิ่ม method payBookingWithWallet, refundBookingToWallet, getTransactions.

Task 4: Customer wallet UI

  • สร้างหน้า customer_wallet_screen.dart แสดง balance, top-up CTA, pending request, transaction history.
  • เพิ่ม navigation จาก profile/account tab.
  • reuse validation amount จาก driver top-up แต่ตัด fallback เติมเงินตรงออก.

Task 5: Checkout integration

  • เพิ่ม wallet payment option ใน food checkout ก่อน แล้วค่อยขยาย ride/parcel.
  • ใช้ final total หลัง coupon เป็นยอดหัก wallet.
  • ถ้า balance ไม่พอ ให้ disable confirm และเสนอปุ่มเติมเงิน.
  • หลัง booking สำเร็จต้อง refresh balance และ transaction history.

Task 6: Admin Web audit

  • เพิ่ม filter role/customer ใน topup requests panel.
  • แสดง wallet transaction related booking และ customer name.
  • เพิ่ม admin adjustment reason required สำหรับ customer wallet.

9. Risks / Guardrails

ยอดเงิน drift จาก coupon

ต้องใช้ยอด final total เดียวกับ checkout และบันทึก coupon_usages.discount_amount หลัง booking ตาม decision เดิม. ถ้า coupon usage fail ต้อง refund wallet.

Race condition / double tap

ห้ามหักเงินใน client. RPC ต้อง lock wallet row และกัน duplicate ด้วย booking_id หรือ wallet_transaction_id.

Top-up proof/payment gateway

PromptPay/manual approval เป็น Phase 1. ถ้าจะต่อ Omise/Stripe/QR auto verify ต้องทำแผน payment gateway แยก และห้ามเดา credential.

Existing driver wallet wording

WalletService ปัจจุบันมี driver-specific behavior. ต้องแยก generic customer path โดยไม่ทำให้ driver wallet/commission regression.

10. Rollout Plan

  1. Phase 0: Review plan กับผู้ใช้และยืนยัน policy: min/max top-up, refund rules, service types ที่เปิดใช้.
  2. Phase 1: Database/RPC + customer wallet screen + manual top-up approval.
  3. Phase 2: เปิด wallet payment เฉพาะ food checkout พร้อม test ครบ.
  4. Phase 3: ขยาย ride/parcel/laundry หลังยอด food flow stable.
  5. Phase 4: เพิ่ม payment gateway auto verification ถ้ามี credential/provider พร้อม.

Stop Conditions

  • ถ้าไม่พบ schema จริงของ wallets หรือ RPC ปัจจุบันไม่ตรง migration history ต้องหยุดเพื่อ audit database ก่อน.
  • ถ้า Supabase credential/local DB ไม่พร้อม ให้ทำได้แค่ static plan/code และรายงาน blocker.
  • ก่อน build หรือ commit ต้อง bump version ตาม AGENTS.md.
  • หลัง code/config change ต้องเรียก code_reviewer ก่อนสรุปว่าเสร็จสมบูรณ์.
