Implementation Plan

แผนงานการแก้ไขข้อบกพร่องระบบ (v1)

รายละเอียดการเปลี่ยนแปลงเพื่อแก้ไขจุดบกพร่อง 11 ประเด็นในโครงการ Jedechai Delivery

เมนูแผนงาน

1. ภาพรวมการดำเนินงาน (Execution Overview)

เอกสารนี้กำหนดรายละเอียดแผนการดำเนินงานแบบ Visual สำหรับการแก้ไข 11 ประเด็นปัญหาที่พบในโค้ดเบสของ Jedechai Delivery ซึ่งได้รับอนุมัติจากผู้ใช้ เรียบร้อยแล้ว

ขอบเขตการแก้ไขจะแบ่งออกเป็น 3 ส่วนหลัก ได้แก่:

  • โมดูลคนขับและลูกค้า (Flutter App): ปลดล็อก Flow ขนส่งพัสดุ (Parcel Delivery), กรองประเภทการให้บริการคนขับรถ, คำนวณราคา Fallback, และแก้ปัญหา UI/UX
  • แผงควบคุมระบบ (Admin Web): แก้ไขระบบ Optimistic Lock bypass และ Native dialog แจ้งเตือนทับซ้อน
  • บริการหลังบ้าน (Supabase Edge Function): เพิ่มระบบจำกัดปริมาณและดักสแปม (Rate Limiting)

2. การแก้ไขระดับวิกฤต (Critical Remediation)

ISSUE-1: โค้ดยืนยันการส่งพัสดุไม่ทำงาน (Driver Confirmation Dead Code) Critical

แผนการแก้ไข:

ในไฟล์ driver_navigation_screen.dart ในเมธอด _handleActionPress() สำหรับออเดอร์ประเภท parcel เมื่อมีการเข้าสู่ส่วน confirmationType == 'delivery' (ซึ่งสถานะปัจจุบันคือ in_transit) โค้ดเดิมมีการไปเรียก _parcelService.updateInTransit(_booking!.id) ซ้ำซ้อน ซึ่งฟังก์ชันดังกล่าวต้องการสเตตัส picked_up ทำให้สืบค้นไม่พบเรคอร์ดและส่งกลับค่า false ส่งผลให้เกิด SnackBar ข้อผิดพลาด และ Return ตัดสิทธิ์ไม่สามารถเข้าสู่หน้าจอถ่ายภาพและเซ็นชื่อยืนยันการจัดส่งได้

ทางทีมงานจะทำการ ลบบล็อกตรวจสอบนี้ออก เนื่องจากสถานะในระบบคลาวด์ได้เป็น in_transit ตั้งแต่เสร็จสิ้นขั้นตอน pickup อยู่แล้ว

// ลบโค้ดส่วนนี้ออกในไฟล์ driver_navigation_screen.dart
if (confirmationType == 'delivery') {
  final parcelReady = await _parcelService.updateInTransit(_booking!.id);
  if (!parcelReady) {
    _showErrorSnackBar(...);
    return;
  }
}
ISSUE-2: ระบบล็อคการรับออเดอร์แทนร้านค้าถูกข้าม (Optimistic Lock Bypass in Admin Web) Critical

แผนการแก้ไข:

ในสคริปต์ app.legacy.js ของ Admin Web ซึ่งรันในโหมด Classic Script มีการกำหนดตัวแปร const ADMIN_MERCHANT_ACCEPT_STATUSES = [...] ทำให้ค่านี้ถูกบีบขอบเขตแบบ Lexical Scope และไม่ได้ถูกบันทึกลงตัวแปร Global ส่งผลให้โมดูลสคริปต์อื่น เช่น ordersActions.js ดึงสเตตัสไปเช็คป้องกันการกดซ้ำ (Race Condition) แล้วได้ค่า undefined และระบบยกเลิกฟิลเตอร์ความปลอดภัยนั้นไปโดยปริยาย

จะแก้ไขโดยผูกตัวแปรเข้ากับ globalThis ตรงๆ เพื่อแชร์ข้อมูลข้ามสคริปต์ได้อย่างเสถียร:

// ปรับปรุงใน app.legacy.js
globalThis.ADMIN_MERCHANT_ACCEPT_STATUSES = ['preparing', 'ready_for_pickup', ...];
ISSUE-3: ระบบส่งอีเมลแจ้งเตือนไม่มี Rate-Limit (Email Spam vulnerability) Critical

แผนการแก้ไข:

ในไฟล์ supabase/functions/send-admin-email/index.ts จะเพิ่มการตรวจสอบ Rate Limiting ในระดับการรับส่งข้อความ และตรวจสอบความถูกต้องของสิทธิ์ผู้ใช้ก่อนการทำงานจัดส่งอีเมลจริง เพื่อลดโอกาสการถูกสแปมหรือค่าบริการอีเมลที่ผิดปกติ

3. การแก้ไขระดับกลาง-สูง (High & Medium Remediation)

ISSUE-4: การคำนวณจำนวนคนขับไม่แยกประเภท (Driver Service Type Filter Missing) High

แผนการแก้ไข:

ในไฟล์ parcel_service_screen.dart ส่วนคำสั่งดึงพิกัดคนขับรอบตัวเพื่อแสดงลูกค้า (_checkNearbyOnlineDrivers) จะเพิ่มฟิลเตอร์คัดกรอง service_type = 'parcel' เพื่อให้แสดงจำนวนเฉพาะคนขับรถที่ให้บริการขนส่งพัสดุได้จริงเท่านั้น

ISSUE-6: การสร้าง Booking พัสดุราคา ฿0 เมื่อมี Fallback (Zero Price Creation) Medium

แผนการแก้ไข:

ปรับปรุงใน parcel_service_screen.dart หากระยะทางไม่สามารถคำนวณได้และระบบทำการ Fallback ไปใช้ 5.0 กม. แทน จะต้องบังคับสั่งคำนวณราคาสุทธิใหม่ (_calculatePrice()) ทันที และเพิ่มเงื่อนไขปัดตก (Reject) หากราคาผลลัพธ์เป็น 0.0 บาท

ISSUE-7: หน้าจอติดตามสถานะพัสดุแสดง "Unknown status" (arrived Status missing) Medium

แผนการแก้ไข:

ในไฟล์ tracking_screen.dart ใน switch-case ของฟังก์ชัน _getStatusInfo และ _getTimelineSteps จะเพิ่มตัวเลือก case 'arrived': เพื่อรองรับเมื่อคนขับเดินทางไปถึงจุดรับของ เพื่อลดความสับสนของลูกค้า

ISSUE-8: ไฟล์รูปถ่ายพัสดุค้างใน Storage (Orphaned Storage Cleanup) Medium

แผนการแก้ไข:

ใน parcel_service_screen.dart เมื่อลูกค้าจองพัสดุและอัปโหลดรูปถ่ายเสร็จเรียบร้อย แต่หากเกิดข้อผิดพลาดในการเรียก RPC หรือบันทึก Booking ลง DB ในขั้นตอนสุดท้าย ระบบจะทำงานลบรูปถ่ายที่อัปโหลดไปแล้วออกจาก Storage ทันทีเพื่อความสะอาดของระบบจัดเก็บ

ISSUE-9: หน้าแอดมินแจ้งเตือนยืนยันซ้ำซ้อนสองชั้น (Double native confirm() dialog) Medium

แผนการแก้ไข:

ในไฟล์ ordersActions.js ของ Admin Web ทำการค้นหาและนำ confirm() ตัวเก่าออก เพื่อใช้งานเพียงแค่ Modal สวยงามที่ตกแต่งเรียบร้อยแล้วเท่านั้น

4. การแก้ไขทั่วไป (Low Remediation)

ISSUE-10: คลีนฟังก์ชันที่ไม่ได้ใช้งานใน ParcelService (Dead Code verification) Low

แผนการแก้ไข:

ในไฟล์ parcel_service.dart สำหรับฟังก์ชัน updateParcelPhoto() ที่ถูกเขียนทิ้งไว้แต่ไม่มี Client UI เรียกใช้ จะทำการปรับปรุงให้มีความรัดกุมในการเช็คสถานะ created เพื่อป้องกันช่องโหว่การแอบแก้ไขภาพในภายหลัง

ISSUE-11: ขนาดปุ่มกดจัดการในแผงแอดมินเล็กเกินไป (Touch target size) Low

แผนการแก้ไข:

ในไฟล์ CSS ของแผงแอดมิน จะปรับปรุงความสูงขั้นต่ำและ Padding ของปุ่มกดแอคชันใน viewport ขนาดเล็กให้มีขนาดไม่ต่ำกว่า 44px ตามข้อกำหนด Mobile UX/UI สากล

5. แผนการตรวจสอบและทดสอบ (Verification & Testing Plan)

หลังจากดำเนินการตามแผนการแก้ไขเรียบร้อยแล้ว จะต้องรันโปรแกรมตรวจสอบเพื่อป้องกันโค้ดถดถอยดังนี้:

  • Flutter Application Static Check: 
    C:\flutter\bin\cache\dart-sdk\bin\dart.exe analyze
  • Admin Web Automated Tests: 
    npm run test
  • การทดสอบระบบพัสดุด้วยตนเอง (Manual End-to-End Test): จำลองการจองพัสดุโดยลูกค้า → คนขับรับงานและยืนยันการรับ/ส่งสินค้า → ติดตามสถานะของลูกค้า → ตรวจสอบการลบรูปภาพในระบบจัดเก็บหากจองขัดข้อง
