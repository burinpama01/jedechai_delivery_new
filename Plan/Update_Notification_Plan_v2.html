Goal / Architecture / Tech Stack

Goal

ให้แอปตรวจเวอร์ชันปัจจุบันกับ policy จาก Supabase แล้วแสดง alert เมื่อมีเวอร์ชันใหม่ โดย admin สามารถปรับข้อความและระดับการบังคับอัปเดตได้จาก admin-web

Architecture

ใช้ Supabase `system_config.app_update_policy` เป็นแหล่งกลาง, Flutter อ่าน policy ด้วย service เฉพาะ, และใช้ guard widget ครอบ `AuthGate` เพื่อควบคุม dialog ก่อนเข้าหน้าหลัก

Tech Stack

Flutter, Supabase, `package_info_plus`, `url_launcher`, `shared_preferences`, admin-web settings bridge, Node smoke tests, Flutter widget/unit tests

Current Evidence

สิ่งที่มีแล้ว

  • Flutter app อยู่ที่ `jedechai_delivery_new/lib/main.dart` และเข้า `AuthGate` จาก `MaterialApp.home`
  • `pubspec.yaml` มี `package_info_plus`, `url_launcher`, `shared_preferences`, `supabase_flutter` แล้ว
  • admin-web มี pattern บันทึกค่า `system_config` ผ่าน settings bridge อยู่แล้ว
  • มีระบบ `notifications` และ FCM สำหรับ release announcement เสริมได้ภายหลัง

ช่องว่างตอนนี้

  • ยังไม่มี `version_check_service.dart` หรือ `app_update_policy_service.dart` ใน main checkout
  • ยังไม่มี field/policy ที่ระบุ `latest_version` และ `min_supported_version`
  • ยังไม่มี UI admin-web สำหรับตั้งค่า update alert
  • ยังไม่มี test สำหรับ version compare และ force update behavior

Release Policy Flow

1. Admin ตั้งค่ากำหนด latest/min version, mode, message, store URL ใน admin-web
2. Save policyบันทึกลง `system_config.app_update_policy` ผ่าน Supabase
3. App startupFlutter อ่าน policy หลัง Supabase init และก่อนเข้า `AuthGate`
4. Compareเทียบ build number ก่อน แล้ว fallback semver ถ้าจำเป็น
5. Alertแสดง optional dialog หรือ force update blocker ตาม policy

Policy Schema

ใช้ JSONB field เดียวเพื่อลด migration surface และให้ admin-web ปรับข้อความได้โดยไม่ต้อง release แอปใหม่

"enabled": true
"latest_version": "1.5.2", "latest_build": 92
"min_supported_version": "1.5.0", "min_supported_build": 90
"mode": "optional" | "force"
"title_th": "มีเวอร์ชันใหม่", "message_th": "กรุณาอัปเดตเพื่อใช้งานฟีเจอร์ล่าสุด"
"android_url": "https://play.google.com/store/apps/details?id=...", "ios_url": "https://apps.apple.com/app/..."
"target_roles": ["customer", "driver", "merchant", "admin"], "starts_at": null, "ends_at": null
Decision: ใช้ build number เป็นตัวตัดสินหลัก เพราะลดปัญหา semver suffix และสอดคล้องกับ version bump rule ของ Flutter (`x.y.z+build`).

Force update

ต้องอัปเดตแอป

เวอร์ชันนี้ไม่รองรับการใช้งานต่อแล้ว กรุณาอัปเดตก่อนเข้าแอป

Optional Update

  • แสดงเมื่อ `current_build < latest_build`
  • ผู้ใช้กด “ไว้ภายหลัง” ได้
  • จำ dismiss ต่อ target version ด้วย `shared_preferences`
  • ไม่ block emergency workflow เช่น driver งานค้าง

Force Update

  • แสดงเมื่อ `current_build < min_supported_build` หรือ `mode=force`
  • dialog ปิดไม่ได้และ back ไม่ออก
  • มีปุ่มเปิด Play Store/App Store เท่านั้น
  • ถ้าเปิด store fail ให้แสดง URL แบบ copy ได้

Failure Behavior

  • ถ้า network/config fail ให้เข้าแอปได้ตามปกติ
  • cache policy ล่าสุดไว้ แต่ไม่ใช้ cache เพื่อ force หากหมดอายุ
  • log debug เฉพาะ non-secret info

Admin Web

  • เพิ่ม card “App Update Policy” ใน settings
  • มี preview ข้อความ optional/force
  • มีปุ่ม save, test parse, และ copy current policy

Rollout Strategy

Stage 1

Disabled deploy

deploy schema + code โดย `enabled=false` เพื่อยืนยันว่า app เดิมเข้าได้และไม่มี alert แปลก

Stage 2

Optional update

เปิด optional alert สำหรับเวอร์ชันใหม่ 24-72 ชั่วโมง ติดตาม feedback และ crash/runtime report

Stage 3

Force update

ใช้เฉพาะเมื่อ backend contract เปลี่ยน, security issue, payment/wallet risk, หรือเวอร์ชันเก่าทำข้อมูลเสียหาย

Risks / Guardrails

Risk Impact Mitigation
ตั้ง force update ผิด ผู้ใช้เข้าแอปไม่ได้ admin-web ต้องมี preview + confirm, และ policy default เป็น disabled
network fetch fail ตอนเปิดแอป เกิด false block fetch fail ต้อง fail-open ยกเว้นมี fresh force policy ที่ valid และยังไม่หมดอายุ
version compare ผิดจาก semver alert ไม่ขึ้นหรือขึ้นผิด ใช้ build number เป็น primary, เพิ่ม unit tests สำหรับ edge cases
Store URL ไม่ถูก ผู้ใช้อัปเดตไม่ได้ validate URL ใน admin-web และ fallback แสดง URL แบบ copy ได้
บังคับ update ระหว่าง driver มีงานค้าง กระทบ delivery operation ใช้ optional ก่อน, force เฉพาะ critical; พิจารณา grace period ตาม role ใน policy
