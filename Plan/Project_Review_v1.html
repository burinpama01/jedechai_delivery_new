2. สรุปรายการแก้ไขและอัปเกรดระบบล่าสุด

ในช่วงการรีวิวที่ผ่านมา ได้มีการแก้ไขข้อผิดพลาดสำคัญในระบบที่เกี่ยวข้องกับประสิทธิภาพและความน่าเชื่อถือของขั้นตอนธุรกิจ โดยแบ่งออกเป็น 4 หมวดหมู่หลัก:

20 พฤษภาคม 2026
การปรับปรุงเวลาและไทม์โซน (Timezone display/serialization policy)

แก้ไขปัญหาการแสดงผลเวลาและข้อมูลนัดหมายคลาดเคลื่อนอันเนื่องมาจากเครื่องผู้ใช้ประมวลผลเวลา UTC ตรงจาก DB ต่างกัน

  • เพิ่มคลาส AppTime ใน lib/common/utils/app_time.dart ทำหน้าที่แปลงเวลาเป็น Asia/Bangkok เสมอ
  • ใช้ Date Key (yyyy-MM-dd) สำหรับกรองสถิติในระดับธุรกิจเพื่อหลีกเลี่ยงผลต่างชั่วโมงในระดับเครื่องมือถือ
  • แปลงเวลาการจอง (Scheduled Order) เป็น UTC ISO ที่สม่ำเสมอก่อนส่งเข้า API/DB
20 พฤษภาคม 2026
แก้ไขระบบการขอลบบัญชีถูกปฏิเสธแล้วค้างระบบ (Account Deletion Unlock Flow)

แก้ไขปัญหาร้ายแรงที่เกิดจากการขอลบบัญชี (Deletion requests) ที่เมื่อแอดมินกดปฏิเสธ แต่สถานะผู้ใช้ยังถูกล็อกอยู่

  • สร้าง database migration 20260520120000_reject_account_deletion_guarded.sql
  • ย้าย logic การปลดล็อกเข้า Database RPC reject_account_deletion_guarded เพื่ออัปเดตแบบ Transaction
  • ให้ Edge Function ทำหน้าที่ดำเนินการผ่าน service_role เพื่อให้มีสิทธิ์แก้ไขข้ามตารางผู้ใช้และหลีกเลี่ยง RLS
  • ทำ Mobile self-heal และดึงสถานะจริงหลังการปฏิเสธ
20 พฤษภาคม 2026
การเชื่อมต่อคูปองส่วนลดของบริการ Ride / Parcel และระบบ Rollback

ขยายความสามารถในการใช้คูปองส่วนลดในฟังก์ชันการเรียกรถ (Ride) และการส่งพัสดุ (Parcel)

  • เชื่อมต่อ CouponEntryWidget เข้ากับหน้าจอเรียกรถและคีย์รับส่งพัสดุ
  • หักมูลค่าส่วนลดจริงและแสดงราคาขีดฆ่าให้เห็นแบบ Visual
  • เพิ่ม Logic การบันทึกใช้งานคูปองด้วย recordUsage หากล้มเหลวจะทำลายการจองอัตโนมัติ (Booking Cancel Rollback)
19 พฤษภาคม 2026
การปรับปรุงความสอดคล้องของเว็บแอดมิน (Admin Web Optimization)

แก้ไขปัญหาที่แอดมินกดรับรายการแทนร้านค้าแล้ว optimistic lock ถูกบายพาส

  • เปิดเผย ADMIN_MERCHANT_ACCEPT_STATUSES ให้กลายเป็น global variable ทั่วทั้งเว็บแอปพลิเคชัน
  • ปรับขนาดปุ่มดำเนินการ (Action button) บนโทรศัพท์มือถือให้มีขนาดไม่ต่ำกว่า 44px ป้องกันการกดพลาด
  • ถอด native confirm popup ที่ซ้ำซ้อนกับ modal ดำเนินการออกไป

3. สถาปัตยกรรมระบบและความปลอดภัย

ระบบ Jedechai Delivery มีการควบคุมและส่งต่อข้อมูลที่ยึดหลัก "สิทธิ์แบบปลอดภัยระดับแถวข้อมูล" (Row Level Security - RLS) และการเขียนโปรแกรมควบคุมธุรกรรมแบบ Atomic ผ่าน RPC ใน PostgreSQL

💡 รูปแบบ Transaction Security
เนื่องจาก RLS มีนโยบายให้ผู้ใช้ทั่วไป (ลูกค้า/คนขับ) แก้ไขข้อมูลได้เฉพาะแถวของตนเองเท่านั้น การทำงานพิเศษของแอดมินที่ต้องตรวจสอบหรือปรับแต่งข้อมูลผู้ใช้จึงต้องย้ายไปทำงานระดับ Server-side (เช่น RPC หรือ Edge Functions) เพื่อหลีกเลี่ยงการถูกปฏิเสธสิทธิ์

กรณีตัวอย่าง: ปฏิเสธคำขอลบบัญชีแบบ Atomic

เมื่อแอดมินสั่งปฏิเสธ แถวข้อมูลในตาราง account_deletion_requests และคอลัมน์ deletion_status ในตาราง profiles จะถูกปรับพร้อมกันในฟังก์ชันเดียว:

CREATE OR REPLACE FUNCTION private.reject_account_deletion_guarded(
  p_request_id TEXT,
  p_admin_id UUID,
  p_reason TEXT
) RETURNS VOID AS $$
BEGIN
  -- 1. อัปเดตตารางคำขอ
  UPDATE public.account_deletion_requests
  SET status = 'rejected',
      reviewed_at = timezone('utc', now()),
      reviewed_by = p_admin_id,
      rejection_reason = p_reason
  WHERE id = p_request_id::BIGINT;

  -- 2. เคลียร์สถานะในโปรไฟล์ผู้ใช้
  UPDATE public.profiles
  SET deletion_status = NULL
  WHERE id = (
    SELECT user_id FROM public.account_deletion_requests WHERE id = p_request_id::BIGINT
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

4. ความเสี่ยงทางเทคนิคที่ยังคงเหลืออยู่ (Residual Technical Risks)

จากการรีวิวซอร์สโค้ดและสภาพแวดล้อมระบบ พบประเด็นความเสี่ยงที่ทีมงานจำเป็นต้องให้ความสำคัญก่อนทำการปล่อยสู่ Production:

🚨 Blocked: Supabase Production Deployment
รายละเอียด: โค้ด Migration (SQL) และฟังก์ชันแอดมินตัวใหม่ถูกแก้ไขและยืนยันในระดับ Local แล้ว แต่การรันคำสั่ง supabase db push เพื่ออัปโหลดขึ้นเซิร์ฟเวอร์หลักยังคงทำไม่ได้ เนื่องจาก Supabase CLI ใน Sandbox ไม่มี credentials (Access Token/Session) ในการล็อกอิน
⚠️ Warning: CLI Toolchain Timeout & Direct SDK Workaround
รายละเอียด: คอมมานด์ไลน์ปกติอย่าง flutter analyze และ flutter test เกิดอาการค้างและหมดเวลา (Timeout) ภายในแซนด์บ็อกซ์
วิธีแก้ไขเฉพาะหน้า: ต้องเรียกใช้ตัวแปลภาษา Dart โดยตรงผ่าน C:\flutter\bin\cache\dart-sdk\bin\dart.exe analyze จึงจะทำงานได้สำเร็จ
📝 ข้อมูลความเสี่ยงและแผนการลดความเสี่ยง
ประเด็นความเสี่ยง ระดับความรุนแรง แนวทางการจัดการความเสี่ยง (Mitigation Plan)
Database Migrations ยังไม่รันบน DB จริง สูงมาก ขอ Access token จากผู้พัฒนาหรือแอดมิน เพื่อรัน dry-run หรือนำคำสั่ง SQL ในโฟลเดอร์ migration ไปติดตั้งบนแดชบอร์ด Supabase ด้วยตนเอง
ความเข้ากันได้ของ Zone เวลาในฝั่งแอปโมบายล์ ปานกลาง หลังจากติดตั้งแอปพลิเคชันเวอร์ชันใหม่ ให้ทดสอบเปลี่ยนเขตเวลาเครื่องมือถือเป็นโซนอื่น (เช่น UTC+0 หรือ UTC+9) และทดสอบจองล่วงหน้าเพื่อยืนยันพฤติกรรม
สิทธิ์ RLS ของ Edge Functions ต่ำ มีการหุ้มสิทธิ์ด้วย schema private และเรียกใช้ service_role ใน backend เรียบร้อยแล้ว ถือว่าผ่านเกณฑ์ความปลอดภัย

5. คำแนะนำและแผนงานในอนาคต (Next Recommendations)

เพื่อนำระบบ Jedechai Delivery ขึ้นใช้งานอย่างสมบูรณ์ ขอแนะนำขั้นตอนการดำเนินการดังต่อไปนี้ตามลำดับความสำคัญ:

  1. การปรับปรุงเวอร์ชันและสร้าง Release Candidate (RC):

    ขยับเวอร์ชันแอปในไฟล์ pubspec.yaml และ package.json ตามกฎของโปรเจกต์ จากนั้นทำคอมมิตระบุหมายเลขเวอร์ชันให้ชัดเจน

  2. การทำ Database Migration Deployment:

    ดำเนินการนำเนื้อหาจาก supabase/migrations/20260520120000_reject_account_deletion_guarded.sql ไปติดตั้งบนฐานข้อมูลหลัก

  3. การรันและดีพลอยแอดมินเว็บพอร์ทัล (Deploy Admin Web):

    อัปโหลดหน้าจอแอดมินเว็บไปยัง Vercel หรือ Netlify พร้อมตั้งค่าไฟล์ config.production.js แยกจากตัวซอร์สโค้ดเพื่อความปลอดภัย

  4. การทดสอบระบบปลายทาง (End-to-End Testing):

    เน้นย้ำการทดสอบสามสถานการณ์หลัก:
    • ลูกค้ากดยกเลิกจองเมื่อเกิดข้อผิดพลาดของระบบคูปอง
    • แอดมินปฏิเสธคำขอลบบัญชี แล้วแอปมือถือลูกค้าต้องแจ้งเตือนปลดล็อกและใช้งานต่อได้ทันที
    • หน้าจอการเงินของคนขับแสดงรายได้รวมในแต่ละวันสอดคล้องกับเวลาไทย UTC+7
