0. อะไรเปลี่ยนจาก v1

Decision ใหม่ (ผู้ใช้ยืนยัน): กดสถานะออเดอร์ได้ ทั้งสองฝั่ง — ทั้งในแอป JDC (ร้าน) และในหน้า StoreOS POS (พนักงาน) แล้ว sync หากันแบบ real-time
หัวข้อv1v2 (ฉบับนี้)
ทิศสถานะJDC → StoreOS อย่างเดียวสองทาง JDC ↔ StoreOS
Edge Fn connect_update_order_statusoptionalจำเป็น (StoreOS→JDC)
กัน loop/echo—ใหม่ คอลัมน์ status_origin + guard ใน trigger
กันกดชนกันoptimistic เดิมoptimistic + เช็ค transition ถูกทิศ

1. บทบาทของ JDC

JDC = LINE MAN (marketplace + คนขับ) · StoreOS = Wongnai POS (เมนู + ครัว) · เชื่อมผ่าน StoreOS Connect แบบ Contract API + Webhook

JDC ต้องทำ:
  • รับเมนู จาก StoreOS → upsert menu_items
  • ส่งออเดอร์ใหม่ ออกไป StoreOS (trigger)
  • ส่งสถานะ เมื่อร้านกดในแอป JDC → StoreOS (trigger)
  • รับสถานะ เมื่อพนักงานกดใน StoreOS POS → อัปเดต booking (Edge Fn) ใหม่ v2
  • รับเปิด/ปิดร้าน จาก StoreOS (Edge Fn)
business logic หลัก (จับคู่คนขับ, จ่ายเงินร้าน, RPC create_food_order/mark_food_ready_guarded) ไม่ต้องแก้

2. โมเดลเงิน — ไม่กระทบ JDC

เงินไม่ไหลผ่าน StoreOS · JDC จ่ายเงินร้านเองตาม flow เดิม:

รับออเดอร์ → อาหารพร้อม → ไรเดอร์รับ → JDC จ่ายร้าน → จบ

StoreOS แค่บันทึกออเดอร์เป็น "ชำระโดย JDC" — ไม่มี settlement/GP ใน StoreOS

3. สถานะสองทาง + กลไกกัน loop (หัวใจของ v2)

ปัญหาเมื่อทำสองทาง: StoreOS อัปเดต JDC → trigger ของ JDC ยิงสถานะกลับ StoreOS → StoreOS อัปเดตอีก → วน infinite loop

วิธีแก้: ติด "ต้นทาง" ของการเปลี่ยนสถานะ

1
เพิ่มคอลัมน์ bookings.status_origin ('jdc' | 'storeos')
2
เมื่อ ร้านกดในแอป JDC → เขียน status พร้อม status_origin='jdc' → trigger ยิงออก StoreOS
3
เมื่อ StoreOS กดผ่าน Edge Fn → เขียน status พร้อม status_origin='storeos' → trigger เห็นว่า origin=storeos → ไม่ยิงกลับ (กัน echo)
4
ฝั่ง StoreOS ก็ทำเหมือนกัน: ถ้าสถานะมาจาก webhook ของ JDC → อัปเดต connect_orders โดยไม่ push กลับ
กฎทอง: "ใครเป็นคนทำให้เปลี่ยน คนนั้นไม่ต้องบอกตัวเอง" — แต่ละฝั่ง emit เฉพาะการเปลี่ยนที่เกิดจากผู้ใช้ฝั่งตัวเองเท่านั้น
เสริม idempotency: ถ้า status ที่รับมา = status ปัจจุบันอยู่แล้ว → no-op (กันกรณี race/ส่งซ้ำ)

4. สรุปสิ่งที่ต้องแก้ฝั่ง JDC

กลุ่มที่อยู่งาน
DBmigrations/menu_items.external_ref/source+index; bookings.status_origin v2 + (opt)pos_order_id; ตาราง pos_connections; เปิด pg_net+pgcrypto

5. DB Migrations

5.1 menu_items รองรับ upsert จาก StoreOS

alter table menu_items add column if not exists external_ref text;   -- products.id ฝั่ง StoreOS
alter table menu_items add column if not exists source text not null default 'jdc';
create unique index if not exists uq_menu_items_merchant_extref
  on menu_items(merchant_id, external_ref) where external_ref is not null;

5.2 bookings: คอลัมน์กัน loop ใหม่ v2

alter table bookings add column if not exists status_origin text;  -- 'jdc' | 'storeos'
alter table bookings add column if not exists pos_order_id text;   -- (option) order id ฝั่ง StoreOS

5.3 ตารางผูกร้าน + secret

create table if not exists pos_connections (
  id uuid primary key default gen_random_uuid(),
  merchant_id uuid not null references profiles(id) on delete cascade,
  provider text not null default 'storeos',
  store_external_id text,
  webhook_secret text not null,            -- shared HMAC secret
  storeos_webhook_url text not null,       -- ปลายทางรับ order/สถานะฝั่ง StoreOS
  menu_managed_by_pos boolean not null default true,
  status text not null default 'active',
  created_at timestamptz not null default now(),
  unique (merchant_id, provider)
);
alter table pos_connections enable row level security;
create policy "merchant reads own pos connection" on pos_connections
  for select using (auth.uid() = merchant_id);
-- เขียน secret = service role/admin เท่านั้น

5.4 extensions

create extension if not exists pg_net;
create extension if not exists pgcrypto;

6. Edge Functions (supabase/functions/ สร้างใหม่)

ทุกตัว verify HMAC ก่อน (secret จาก pos_connections ตาม merchant)

6.1 connect_upsert_menu StoreOS เรียก

// body: { merchant_id, items:[{external_ref,name,description,price,image_url,
//          is_available,category,preparation_time}], full_sync:false }
// verify HMAC → upsert onConflict(merchant_id,external_ref), source='storeos'
// full_sync=true → ปิด is_available เมนู source='storeos' ที่ไม่อยู่ใน payload
// → 200 { upserted:n }

6.2 connect_set_shop_status StoreOS เรียก

// body: { merchant_id, is_open }
// verify HMAC → update profiles set shop_status=is_open, is_online=is_open

6.3 connect_update_order_status StoreOS เรียก จำเป็น v2

// body: { booking_id, status }   // status (JDC term) = preparing | ready_for_pickup | cancelled
// 1) verify HMAC
// 2) เช็ค transition ถูกทิศ (กันกดถอยหลัง/ชนกับแอป) — reuse logic mark_food_ready_guarded ถ้าได้
// 3) update bookings set status=:status, status_origin='storeos'   ← สำคัญ! ติดธงกัน loop
//      where id=:booking_id and status = :expected_current   (optimistic guard)
// 4) ผลลัพธ์: realtime stream ของแอป merchant เห็น update ทันที (ไม่ต้องแก้ Flutter)
// → 200 { ok:true, status } | 409 { conflict, current }
ต้องเซ็ต status_origin='storeos' ทุกครั้ง ไม่งั้น trigger §7 จะยิงสถานะกลับ StoreOS เกิด loop

7. ส่งออเดอร์ + สถานะออก StoreOS (Webhook ขาออก + guard)

trigger บน bookings(food) → คำนวณ HMAC (pgcrypto) → ยิง async (pg_net) · ข้ามถ้า origin = storeos

create or replace function notify_storeos_order() returns trigger
language plpgsql security definer as $$
declare v_conn pos_connections%rowtype; v_body jsonb; v_sig text; v_topic text;
begin
  if new.service_type <> 'food' then return new; end if;

  -- ★ กัน loop: ถ้าการเปลี่ยนนี้มาจาก StoreOS เอง ไม่ต้องยิงกลับ
  if tg_op='UPDATE' and new.status_origin = 'storeos'
     and new.status is distinct from old.status then
    return new;
  end if;
  -- กัน no-op: UPDATE ที่ status ไม่เปลี่ยน ไม่ต้องยิง
  if tg_op='UPDATE' and new.status is not distinct from old.status then
    return new;
  end if;

  select * into v_conn from pos_connections
    where merchant_id = new.merchant_id and status='active' limit 1;
  if not found then return new; end if;

  v_topic := case when tg_op='INSERT' then 'order.created' else 'order.status' end;
  v_body := jsonb_build_object(
    'topic', v_topic, 'booking_id', new.id, 'merchant_id', new.merchant_id,
    'status', new.status, 'total', new.price, 'paid', true,
    'ts', extract(epoch from now())::bigint
    -- INSERT: เพิ่ม items (join booking_items), customer, dropoff
  );
  v_sig := encode(hmac(v_body::text, v_conn.webhook_secret, 'sha256'), 'hex');
  perform net.http_post(
    url := v_conn.storeos_webhook_url,
    headers := jsonb_build_object('Content-Type','application/json',
               'X-Connect-Signature','sha256='||v_sig),
    body := v_body);
  return new;
end $$;

create trigger trg_notify_storeos_order
after insert or update of status on bookings
for each row execute function notify_storeos_order();
เมื่อร้านกดในแอป JDC ผ่าน acceptOrder/markFoodReady → ควรเซ็ต status_origin='jdc' ด้วย (ดู §9.2) เพื่อให้ trigger ยิงออก; ส่วน Edge Fn StoreOS เซ็ต 'storeos' เพื่อไม่ยิงกลับ

8. ตาราง map สถานะ (สองทาง)

StoreOS fulfillment⇄JDC booking statusผล
received⇄pending_merchantรอร้านรับ
accepted / preparing⇄preparingร้านรับ/กำลังทำ → หาคนขับ
ready⇄ready_for_pickupคนขับเข้ารับ
completed⇄in_transit → completedJDC คุมต่อถึงลูกค้า
cancelled⇄cancelledยกเลิก (มีกติกาใครยกเลิกได้เมื่อไร)
ข้อควรระวังสถานะที่คนขับเป็นเจ้าของ: driver_accepted, matched, arrived_at_merchant, in_transit เป็นของฝั่ง JDC/คนขับ — StoreOS ห้ามสั่ง set สถานะกลุ่มนี้ผ่าน Edge Fn (อนุญาตเฉพาะ preparing/ready_for_pickup/cancelled)

9. แก้ Flutter UI

9.1 หน้าจัดการเมนู — read-only เมื่อเชื่อม POS

  • โหลด pos_connections; ถ้า menu_managed_by_pos=true → ซ่อน/disable เพิ่ม-แก้-ลบ + ชื่อ/ราคา/รูป
  • แบนเนอร์: "เมนูจัดการจาก StoreOS POS"
  • คงให้ toggle "ของหมดวันนี้" (is_available) ได้

9.2 หน้าออเดอร์ร้าน — แก้เล็กน้อย v2

  • รับสถานะจาก StoreOS: watchOrders เป็น realtime stream จาก bookings อยู่แล้ว → เมื่อ Edge Fn StoreOS อัปเดต booking หน้าจอเด้งเอง ไม่ต้องแก้
  • ส่งสถานะออก: ใน acceptOrder/markFoodReady/finishOrder เพิ่มเขียน status_origin:'jdc' ตอน update เพื่อให้ trigger ยิงออก StoreOS
// merchant_order_service.dart — acceptOrder()
.update({ 'status': newStatus,
          'status_origin': 'jdc',                     // ★ เพิ่มบรรทัดนี้
          'updated_at': DateTime.now().toIso8601String() })
// markFoodReady ใช้ RPC → ปรับ RPC mark_food_ready_guarded ให้ set status_origin='jdc' ด้วย

9.3 หน้าตั้งค่าร้าน — ส่วนเชื่อม POS

  • โชว์ merchant_id (รหัสเชื่อม) + สถานะเชื่อมต่อ + เวลา sync เมนูล่าสุด

13. สัญญา API (อ้างอิงร่วมกับ StoreOS)

JDC → StoreOS   POST {storeos}/api/connect/v1/webhooks/jdc/order
                Body: { topic:'order.created'|'order.status', booking_id, merchant_id,
                        status, items[], customer, dropoff, total, paid:true, ts }

StoreOS → JDC   POST {jdc}/functions/v1/connect_upsert_menu          { merchant_id, items[], full_sync }
StoreOS → JDC   POST {jdc}/functions/v1/connect_set_shop_status      { merchant_id, is_open }
StoreOS → JDC   POST {jdc}/functions/v1/connect_update_order_status  { booking_id, status }   ← v2 จำเป็น

HMAC = hex(hmac_sha256(rawBody, webhook_secret)) · ตรวจ ts ไม่เกิน 5 นาที · ทุก status update ติด status_origin กัน loop
