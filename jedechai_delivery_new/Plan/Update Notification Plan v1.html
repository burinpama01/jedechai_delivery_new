1. เป้าหมายและ Success Criteria

Primary Goal

ให้ผู้ใช้ย้ายไปเวอร์ชันล่าสุดเร็วขึ้น

ลดจำนวนผู้ใช้ที่ยังเปิดแอปเวอร์ชันเก่า โดยสื่อสารให้ชัดว่าเวอร์ชันใหม่มีความเสถียร ปลอดภัย และจำเป็นต่อการใช้งานต่อเนื่อง

Success Criteria
  • ผู้ใช้ active อย่างน้อย 80% อยู่บนเวอร์ชันล่าสุดภายใน 7 วันหลังประกาศ
  • ลด ticket/support เรื่องปัญหาที่แก้แล้วในเวอร์ชันใหม่
  • ไม่มี spike ของ complaint จากข้อความแจ้งเตือนถี่เกินไป
Release Risk

ถ้าใช้ artifact ที่มี build number เดิม อาจชนกับ store release เดิมได้ ก่อนประกาศ public update ควรใช้ build number ใหม่ที่ store ยอมรับแล้ว

2. กลุ่มผู้ใช้และระดับความเร่งด่วน

กลุ่ม ข้อความหลัก ความเร่งด่วน ช่องทางแนะนำ
Customer อัปเดตเพื่อใช้งานสั่งอาหาร ส่งพัสดุ และเรียกรถได้เสถียรขึ้น Medium In-app banner, push notification, LINE OA
Driver อัปเดตเพื่อรับงาน แจ้งเตือน และนำทางได้ถูกต้องขึ้น High In-app modal, push notification, กลุ่ม LINE คนขับ
Merchant อัปเดตเพื่อรับออเดอร์และแจ้งเตือนร้านค้าได้ครบถ้วน High In-app modal, merchant LINE, direct admin call สำหรับร้านสำคัญ
Admin อัปเดตเพื่อใช้ flow ล่าสุดและลด mismatch กับ backend High Internal chat, checklist ก่อนปฏิบัติงาน

3. กลยุทธ์การแจ้งเตือนแบบ 3 ระดับ

Level 1: Soft Notice

แจ้งแบบไม่บังคับ

  • แสดง banner ในหน้าแรก
  • ปุ่มหลัก: “อัปเดตตอนนี้”
  • ปุ่มรอง: “ภายหลัง”
  • เหมาะกับ release ปกติหรือปรับปรุง UX
Level 2: Strong Reminder

แจ้งแบบเด่นชัด

  • แสดง modal เมื่อเปิดแอป
  • ให้เลื่อนการอัปเดตได้ไม่เกิน 24 ชั่วโมง
  • เหมาะกับ bugfix สำคัญ เช่น notification, payment, order status
Level 3: Force Update

บังคับอัปเดต

  • ล็อก flow หลักจนกว่าจะอัปเดต
  • ใช้เฉพาะกรณี critical เช่น security, data loss, payment, incompatible API
  • ต้องมี min supported version จาก remote config/backend

4. Timeline การสื่อสาร

T-1 วัน

เตรียมประกาศและ support script

  • ยืนยัน version ล่าสุดบน Play Store/App Store หรือ distribution channel
  • เตรียมข้อความสั้น, ข้อความยาว, FAQ และคำตอบสำหรับ support
  • เช็ก deep link ไปหน้า store ให้ถูกต้อง
วันปล่อย

เริ่ม soft notice และประกาศช่องทางหลัก

  • เปิด in-app banner สำหรับทุก role
  • ส่ง LINE OA หรือประกาศในกลุ่มคนขับ/ร้านค้า
  • ส่ง push รอบแรกเฉพาะผู้ใช้ที่ยังไม่ใช่เวอร์ชันล่าสุด ถ้ามี version tracking
+24 ชั่วโมง

ยกระดับสำหรับกลุ่มสำคัญ

  • แสดง modal สำหรับ Driver และ Merchant ที่ยังไม่อัปเดต
  • Admin ติดตามร้าน/คนขับ active สูงที่ยังไม่อัปเดต
  • ดู metric crash, complaint, notification delivery
+72 ชั่วโมง

พิจารณา force update เฉพาะกรณีจำเป็น

  • ถ้าเวอร์ชันเก่ากระทบ order/payment/notification ให้ตั้ง min supported version
  • ถ้าเป็น release ปกติ ให้คง reminder แบบไม่บังคับและลดความถี่

5. ข้อความที่ใช้สื่อสาร

In-app Banner
มีแอปเวอร์ชันใหม่แล้ว อัปเดตเพื่อให้การรับแจ้งเตือน ออเดอร์ และการใช้งานต่าง ๆ เสถียรขึ้น [อัปเดตตอนนี้] [ภายหลัง]
Push Notification
อัปเดตแอปเจ๊ได๋ชัยเป็นเวอร์ชันล่าสุด เพื่อให้รับออเดอร์และการแจ้งเตือนได้ครบถ้วน แตะเพื่ออัปเดตตอนนี้
LINE OA / Broadcast
เรียนผู้ใช้งานเจ๊ได๋ชัย ขณะนี้มีแอปเวอร์ชันล่าสุดพร้อมใช้งานแล้ว กรุณาอัปเดตแอปเพื่อให้การสั่งซื้อ รับงาน แจ้งเตือน และติดตามสถานะทำงานได้เสถียรขึ้น หากยังไม่อัปเดต บางฟีเจอร์อาจทำงานไม่สมบูรณ์ในอนาคต
Force Update Modal
จำเป็นต้องอัปเดตแอป เวอร์ชันที่คุณใช้อยู่ไม่รองรับการทำงานล่าสุดแล้ว กรุณาอัปเดตเพื่อใช้งานต่อ [อัปเดตแอป]

6. Implementation Plan เชิงระบบ

งาน รายละเอียด Validation
เพิ่ม version tracking เก็บ app version/build number จาก `package_info_plus` ตอนเปิดแอป และ sync ไป backend profile/session ถ้ามีนโยบาย privacy รองรับ เห็นจำนวนผู้ใช้แยกตาม role/version ได้
เพิ่ม remote config กำหนด `latestVersion`, `minSupportedVersion`, `updateUrlAndroid`, `updateUrlIos`, `messageLevel` จาก backend หรือ Supabase table เปลี่ยนระดับ notice ได้โดยไม่ต้องปล่อยแอปใหม่
เพิ่ม update gate UI สร้าง component กลางสำหรับ banner/modal/force screen และใช้ร่วมทุก role Customer, Driver, Merchant, Admin เห็นข้อความถูกตาม role
เพิ่ม analytics บันทึก event เช่น `update_notice_seen`, `update_clicked`, `update_dismissed`, `force_update_shown` วัด conversion จาก notice เป็น update ได้
เตรียม support workflow ทำ FAQ และ script ให้ admin/support ตอบผู้ใช้ที่อัปเดตไม่ได้ Support ตอบเป็นมาตรฐานเดียวกัน

7. Guardrails ก่อนประกาศจริง

  • ยืนยันว่า APK/AAB ที่จะปล่อยมี version/build number ไม่ชนกับ release เดิมบน store
  • ยืนยันว่า store listing เปิดให้ผู้ใช้กลุ่มเป้าหมายเห็น update แล้ว
  • ห้ามส่งข้อความที่บอกว่า “จำเป็นต้องอัปเดต” ถ้ายังไม่ได้ตั้ง force update จริง
  • ถ้าเกี่ยวกับ payment/auth/customer data ต้อง review security และ rollback path ก่อน
  • ต้องมีวิธียกเลิก force update จาก remote config หากพบ regression หลังปล่อย

8. Checklist สำหรับทีม

ก่อนส่งประกาศ

  1. ยืนยัน release artifact และ version ล่าสุด
  2. ยืนยันลิงก์ store หรือ download URL
  3. เลือก notice level ตามความเสี่ยง
  4. ตรวจข้อความไทยให้ชัดและไม่ทำให้ผู้ใช้ตื่นตระหนก
  5. เตรียม FAQ/support response

หลังส่งประกาศ

  1. ติดตาม adoption ตาม role/version
  2. ดู crash/error/support ticket
  3. ส่ง reminder เฉพาะกลุ่มที่ยังไม่อัปเดต
  4. ตัดสินใจว่าจะ force update หรือไม่หลัง 72 ชั่วโมง
  5. บันทึกผลลัพธ์และบทเรียนลง Obsidian
