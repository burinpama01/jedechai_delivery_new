1. บทบาทของ JDC ในระบบเชื่อม

JDC = ฝั่ง LINE MAN (marketplace + คนขับ) · StoreOS = ฝั่ง Wongnai POS (จัดการเมนู + ครัว) · เชื่อมผ่าน StoreOS Connect (โมดูลใน StoreOS) ด้วย Contract API + Webhook

JDC ต้องทำ 4 อย่างเพื่อรับ-ส่งกับ StoreOS:

  • รับเมนู ที่ StoreOS ดันมา → upsert ลง menu_items (Edge Fn)
  • ส่งออเดอร์ใหม่ ออกไปหา StoreOS เมื่อมี booking อาหาร (DB trigger → webhook)
  • ส่งสถานะ เมื่อร้านกดรับ/อาหารพร้อม → ให้ StoreOS เห็น (auto จาก trigger เดียวกัน)
  • รับคำสั่งเปิด/ปิดร้าน จาก StoreOS → เซ็ต profiles.shop_status (Edge Fn)
ข่าวดี: business logic หลักของ JDC (จับคู่คนขับ, จ่ายเงินร้าน, RPC create_food_order, mark_food_ready_guarded) ไม่ต้องแก้ — งานคือ "เปิดประตู" รับ-ส่งกับ StoreOS เท่านั้น

2. โมเดลเงิน (ยืนยันแล้ว) — ไม่กระทบ JDC

เงินไม่ไหลผ่าน StoreOS เลย — JDC จ่ายเงินให้ร้านเองตาม flow เดิม:

ร้านกดรับออเดอร์ → กดอาหารพร้อม → ไรเดอร์กดรับอาหาร → JDC จ่ายเงินร้าน → จบ flow ฝั่งร้าน

  • JDC: settlement / จ่ายเงินร้าน / ค่า GP คงเดิมทั้งหมด ไม่ต้องแก้
  • StoreOS: แค่ บันทึกออเดอร์เป็น "ชำระโดย JDC" เพื่อให้ยอดขาย/ครัวครบ ไม่มีการรับเงินจริงใน StoreOS

3. ใครเป็นคนคุมสถานะออเดอร์ (สำคัญ)

จาก flow ที่คุณอธิบาย ร้านกดปุ่ม ในแอป JDC (รับออเดอร์/อาหารพร้อม) → ดังนั้น MVP ใช้แนวนี้:

✓ แนะนำ MVP: JDC เป็น control surface หลัก

  • ร้านกดรับ/พร้อมในแอป JDC ตามเดิม
  • ทุกครั้งที่ bookings.status เปลี่ยน → trigger ส่งสถานะไป StoreOS อัตโนมัติ (JDC → StoreOS)
  • StoreOS POS/KDS แสดงสถานะ live + พิมพ์ตั๋วครัว + ปิดออเดอร์เมื่อ completed

ทางเลือก (Phase 1c+): StoreOS ก็กดได้

  • ถ้าอยากให้พนักงานกดรับ/พร้อมจาก StoreOS POS ด้วย → เพิ่ม Edge Fn connect_update_order_status (StoreOS → JDC)
  • ต้องกันชนกัน (กดสองที่) ด้วย optimistic concurrency เดิม (.eq('status', current))
ต้องยืนยัน 1 ข้อ: MVP ให้กดที่ JDC อย่างเดียวก่อน (StoreOS แค่ดู) ใช่ไหม? ถ้าใช่ จะตัด Edge Fn update_status ออกจาก MVP ลดงานฝั่ง JDC ลงอีก — นี่ต่างจากแผน StoreOS v1 ที่ระบุ status เป็น StoreOS→JDC (จะปรับเป็น v2)

5. DB Migrations (Supabase project B)

5.1 เพิ่ม external ref ให้ menu_items (รองรับ upsert จาก StoreOS)

alter table menu_items add column if not exists external_ref text;   -- = products.id ฝั่ง StoreOS
alter table menu_items add column if not exists source text not null default 'jdc'; -- 'jdc' | 'storeos'
-- กัน upsert ซ้ำ: 1 สินค้า StoreOS = 1 เมนูต่อร้าน
create unique index if not exists uq_menu_items_merchant_extref
  on menu_items(merchant_id, external_ref) where external_ref is not null;
ราคาในเมนู (menu_items.price) = ราคาเดลิเวอรี ที่ StoreOS ส่งมา (อาจต่างจากราคาหน้าร้าน) — JDC เก็บตามที่ได้รับ ไม่ต้องคำนวณเอง

5.2 ตารางผูกร้านฝั่ง JDC (เก็บ secret + ปลายทาง StoreOS)

create table if not exists pos_connections (
  id              uuid primary key default gen_random_uuid(),
  merchant_id     uuid not null references profiles(id) on delete cascade,
  provider        text not null default 'storeos',
  store_external_id text,                 -- store_id ฝั่ง StoreOS (ไว้อ้างอิง)
  webhook_secret  text not null,          -- shared secret (HMAC) — ตรงกับฝั่ง StoreOS
  storeos_webhook_url text not null,      -- ปลายทางรับออเดอร์ฝั่ง StoreOS
  menu_managed_by_pos boolean not null default true, -- true = เมนู read-only ในแอป
  status          text not null default 'active',
  created_at      timestamptz not null default now(),
  unique (merchant_id, provider)
);
alter table pos_connections enable row level security;
create policy "merchant reads own pos connection" on pos_connections
  for select using (auth.uid() = merchant_id);
-- เขียน/แก้ secret ทำผ่าน service role / admin เท่านั้น (ไม่เปิดให้ client เขียน secret)

5.3 (option) ผูกออเดอร์กับ POS

alter table bookings add column if not exists pos_order_id text;  -- order id ฝั่ง StoreOS เผื่อ debug/ลิงก์

5.4 เปิด extension ที่ต้องใช้

create extension if not exists pg_net;     -- ยิง http จาก trigger
create extension if not exists pgcrypto;   -- คำนวณ hmac() สำหรับเซ็น webhook

6. Edge Functions (สร้างโฟลเดอร์ supabase/functions/ ใหม่)

แต่ละฟังก์ชันต้อง verify HMAC จาก header ก่อนทำงาน (secret ดึงจาก pos_connections ตาม merchant)

6.1 connect_upsert_menu StoreOS เรียก

// POST /functions/v1/connect_upsert_menu
// body: { merchant_id, items:[{ external_ref, name, description, price,
//          image_url, is_available, category, preparation_time }], full_sync:false }
// 1) verify X-Connect-Signature (hmac sha256 ของ body ด้วย pos_connections.webhook_secret)
// 2) upsert ทีละรายการ onConflict (merchant_id, external_ref), source='storeos'
// 3) ถ้า full_sync=true: set is_available=false ให้เมนู source='storeos' ที่ไม่ได้อยู่ใน payload
// → 200 { upserted: n }

6.2 connect_set_shop_status StoreOS เรียก

// POST /functions/v1/connect_set_shop_status
// body: { merchant_id, is_open }
// verify HMAC → update profiles set shop_status=is_open, is_online=is_open
// (ใช้ field เดียวกับ toggleShopStatus เดิม) → 200 { ok:true }

6.3 connect_update_order_status StoreOS เรียก option

// ทำเฉพาะถ้าต้องการให้กดสถานะจาก StoreOS POS ได้ด้วย
// body: { booking_id, status }  // status = preparing | ready_for_pickup | cancelled
// verify HMAC → update bookings ด้วย optimistic guard (กันชนกับการกดในแอป)
// ควร reuse logic เดียวกับ mark_food_ready_guarded เพื่อความสอดคล้อง
ถ้ายืนยันว่า MVP กดที่ JDC อย่างเดียว → ข้าม 6.3 ไปก่อน

7. ส่งออเดอร์ + สถานะออกไป StoreOS (Webhook ขาออก)

ใช้ DB trigger บน bookings (เฉพาะ service_type='food' และร้านที่มี pos_connections) → คำนวณ HMAC ด้วย pgcrypto → ยิง async ด้วย pg_net

1
INSERT booking อาหาร → topic order.created (ส่งทั้งรายการ items + ลูกค้า + ยอด + paid:true)
2
UPDATE สถานะ (preparing/ready_for_pickup/completed/cancelled) → topic order.status
create or replace function notify_storeos_order() returns trigger
language plpgsql security definer as $$
declare
  v_conn pos_connections%rowtype;
  v_body jsonb; v_sig text;
begin
  if new.service_type <> 'food' then return new; end if;
  select * into v_conn from pos_connections
    where merchant_id = new.merchant_id and status='active' limit 1;
  if not found then return new; end if;

  v_body := jsonb_build_object(
    'topic', case when tg_op='INSERT' then 'order.created' else 'order.status' end,
    'booking_id', new.id, 'merchant_id', new.merchant_id,
    'status', new.status, 'total', new.price, 'paid', true,
    'ts', extract(epoch from now())::bigint
    -- INSERT: เพิ่ม items (join booking_items), customer, dropoff
  );
  v_sig := encode(hmac(v_body::text, v_conn.webhook_secret, 'sha256'), 'hex');

  perform net.http_post(
    url := v_conn.storeos_webhook_url,
    headers := jsonb_build_object('Content-Type','application/json',
               'X-Connect-Signature','sha256='||v_sig),
    body := v_body);
  return new;
end $$;

create trigger trg_notify_storeos_order
after insert or update of status on bookings
for each row execute function notify_storeos_order();
กันบล็อกการสร้างออเดอร์: pg_net ทำงาน async อยู่แล้ว (คิว) จึงไม่หน่วงการ insert; ถ้าปลายทางล่ม StoreOS จะอาศัย reconcile/retry ฝั่งตัวเอง — แนะนำเพิ่มตาราง log การยิงไว้ debug ภายหลัง

8. แก้ Flutter UI

8.1 หน้าจัดการเมนู — เมนู read-only เมื่อเชื่อม POS

โหลด pos_connections ของร้าน ถ้า menu_managed_by_pos = true:

  • ซ่อน/disable ปุ่ม เพิ่ม-แก้-ลบ เมนู และช่องแก้ชื่อ/ราคา/รูป
  • โชว์แบนเนอร์: "เมนูจัดการจาก StoreOS POS — แก้ที่ระบบหลังร้าน"
  • คงให้ toggle "ของหมดวันนี้" (is_available) ได้ เพื่อความเร็วหน้าร้าน (จะ sync กลับ optional ภายหลัง)
ป้องกันการแก้สองที่จนข้อมูลขัดกัน — แหล่งความจริงของเมนูคือ StoreOS

8.2 หน้าออเดอร์ร้าน — ไม่ต้องแก้

acceptOrder / markFoodReady / finishOrder ทำงานเดิม → trigger §7 ส่งสถานะให้ StoreOS เอง

8.3 หน้าตั้งค่าร้าน — ส่วนเชื่อมต่อ POS

  • แสดง "รหัสร้านสำหรับเชื่อม POS" = merchant_id (ให้ผู้ใช้ก๊อปไปกรอกฝั่ง StoreOS)
  • แสดงสถานะ: เชื่อมแล้ว/ยังไม่เชื่อม + เวลาที่เมนู sync ล่าสุด
  • (ตามขั้นตอนผูกด้วยมือ §10)

9. ราคา & การเลือกสินค้าขึ้น JDC (ฝั่ง StoreOS เป็นคนคุม)

เรื่องนี้ทำที่ StoreOS ไม่ใช่ JDC — แต่ใส่ไว้ให้เห็นภาพรวม
ต้องมีใน StoreOS (products)ความหมายJDC ได้รับเป็น
available_for_delivery (bool ใหม่)เลือกว่าสินค้านี้จะขึ้น JDC ไหม (sync เฉพาะที่ติ๊ก)มี/ไม่มีใน payload
delivery_price (numeric ใหม่, nullable)ราคาขายบน JDC; ถ้า null ใช้ base_pricemenu_items.price

StoreOS Connect จะดันเฉพาะสินค้าที่ available_for_delivery=true และส่ง delivery_price ?? base_price เป็น price → JDC ไม่ต้องรู้เรื่องราคาหน้าร้านเลย

10. ขั้นตอนผูกร้านด้วยมือ (MVP)

1
ร้านเปิดแอป JDC → หน้าตั้งค่า → ก๊อป merchant_id
2
super-admin StoreOS เปิดหน้า Connect → สร้าง link: กรอก merchant_id, เลือก store, ระบบ generate webhook_secret
3
เอา webhook_secret + storeos_webhook_url ไปบันทึกใน pos_connections ของ JDC (ผ่าน admin/service role — ห้ามให้ client เขียน secret)
4
กด "Sync เมนูทั้งหมด" ที่ StoreOS → menu_items ฝั่ง JDC เต็ม → พร้อมรับออเดอร์
Blocker ความปลอดภัย (ตาม AGENTS.md): secret/URL ต้องตั้งผ่านช่องทางปลอดภัย ไม่ commit ลง repo, ไม่โชว์ใน client. JDC service key ของ StoreOS เก็บใน Vercel env ฝั่ง StoreOS เท่านั้น

11. Checklist + ลำดับการทำฝั่ง JDC

  • [DB] migration: menu_items.external_ref/source + unique index
  • [DB] migration: ตาราง pos_connections + RLS
  • [DB] เปิด extension pg_net + pgcrypto
  • [FN] Edge Fn connect_upsert_menu (+ verify HMAC)
  • [FN] Edge Fn connect_set_shop_status
  • [DB] trigger notify_storeos_order (order.created + order.status)
  • [UI] เมนู read-only เมื่อ menu_managed_by_pos + toggle ของหมด
  • [UI] หน้าตั้งค่า: โชว์ merchant_id + สถานะเชื่อมต่อ
  • [SEC] ใส่ secret ต่อร้านผ่าน service role; test HMAC ทั้งขาเข้า-ออก
  • [opt] Edge Fn connect_update_order_status (ถ้าให้ StoreOS กดสถานะได้)
ลำดับแนะนำ: DB (5) → Edge Functions (6) → trigger ขาออก (7) → ทดสอบ end-to-end กับ StoreOS staging → ค่อยแก้ Flutter UI (8)

12. สัญญา API (อ้างอิงร่วมกับฝั่ง StoreOS)

JDC → StoreOS   POST {storeos}/api/connect/v1/webhooks/jdc/order
                Headers: X-Connect-Signature: sha256=<hmac>
                Body: { topic:'order.created'|'order.status', booking_id, merchant_id,
                        status, items[], customer, dropoff, total, paid:true, ts }

StoreOS → JDC   POST {jdc}/functions/v1/connect_upsert_menu        { merchant_id, items[], full_sync }
StoreOS → JDC   POST {jdc}/functions/v1/connect_set_shop_status    { merchant_id, is_open }
StoreOS → JDC   POST {jdc}/functions/v1/connect_update_order_status { booking_id, status }   (option)

HMAC = hex(hmac_sha256(rawBody, webhook_secret)) · ตรวจ timestamp (ts) ไม่เกิน 5 นาที กัน replay
